## 一、背景与目标

### 1.1 背景

`@lania-pro-components/components` 当前覆盖表单（ProForm）、表格（ProTable）、弹窗（ProDialog）、选择器（ProSelect）、上传（ProUpload）、操作按钮（ActionButton）六大场景，但**数据可视化**这一中后台高频场景仍为空白。

目前各业务项目各自封装一层 echarts / highcharts / antv-g2，存在以下问题：

- **重复封装**：loading/error/empty 三态、resize、主题切换、tooltip 格式化、远程数据加载等模板代码在每个项目重复
- **图表库耦合业务**：业务代码直接 import echarts，切换图表库成本极高；同一项目内不同页面用不同图表库时无统一抽象
- **与现有组件体系割裂**：ProTable 是数据明细视图，ProChart 是数据统计视图，二者本质共享"数据加载 + 状态管理 + 渲染"链路，但目前无法复用
- **性能监控盲区**：图表渲染往往是页面性能瓶颈，但当前图表封装层未接入 `PerformanceMonitor`，难以定位

### 1.2 目标

1. **图表库无关**：通过 **Adapter 模式** 抽象图表实例的创建、更新、销毁生命周期，业务代码不直接依赖任何具体图表库
2. **多图表库共存**：同一项目、同一页面可混用 echarts / highcharts / g2，通过 adapter prop 切换，互不干扰
3. **Schema 与原生配置双形态**：低门槛的 Schema 模式（声明 `type` + `dataSource` + `xField/yField`）+ 高灵活的 Option 模式（直接传 echarts option）
4. **与现有架构无缝衔接**：复用 utils（format/performance）、复用 shared（useAsyncRequest/useCache/useResponsive）、复用 theme（light/dark）
5. **按需加载**：echarts / highcharts 等大体积库通过动态 import + adapter 注册机制实现 Tree Shaking，不引入 ProChart 即不打包

### 1.3 范围

**包含**：

- `packages/components/ProChart/` 新增组件目录
- `packages/shared/` 新增图表相关 hook 与类型（`useChartInstance` / `useResizeObserver` / `ChartAdapter` 类型）
- `packages/components/ProChart/adapters/` 内置 ECharts / Highcharts / G2 三种 adapter（按需 import）
- `packages/components/ProChart/transformers/` Schema 模式的数据转换器

**不包含**：

- 不重新实现任何图表库的渲染能力（adapter 是薄壳，渲染仍由 echarts/highcharts 完成）
- 不实现图表编辑器、图表设计器等重型能力（未来可独立组件）
- 不修改现有 6 个组件的公开 API

## 二、现状与动机

### 2.1 现有可复用底座扫描

| 复用对象 | 来源 | ProChart 用途 |
| -------- | ---- | ------------- |
| `useAsyncRequest`（P1-1 规划） | shared/hooks | 远程数据加载、防抖、取消、缓存 |
| `useCache`（P1-4 规划） | shared/hooks | 数据缓存（同查询条件命中缓存） |
| `useResponsive`（P1-4 规划） | shared/hooks | 容器尺寸变化触发 resize |
| `useVirtualScroll`（P0-2 规划） | shared/hooks | 大数据量散点图的虚拟化（远期） |
| `PerformanceMonitor`（P0-3 规划） | shared/PerformanceMonitor | 图表渲染耗时打点 `chart-render` |
| `formatNumber/formatMoney/formatPercent/formatDate` | utils/format | tooltip / axis / legend 格式化 |
| `deepMerge` | utils/object | 默认 option 与用户 option 合并 |
| `defineEnumMap` | utils/defineEnumMap | 系列名/颜色枚举映射 |
| `ThemeProvider` / `useTheme` | theme | light/dark 切换映射到图表主题 |
| `chartAdapterRegistry`（新增） | ProChart 内部 | adapter 运行时注册与查找 |

**结论**：底层底座完备，ProChart 是上述能力的"组合消费方"，落地后能反向验证 shared 抽象的合理性。

### 2.2 图表库适配难点分析

| 维度 | echarts | highcharts | antv-g2 |
| ---- | ------- | ---------- | ------- |
| 实例创建 | `echarts.init(dom)` | `Highcharts.chart(dom, options)` | `new Chart({ container, options })` |
| 配置更新 | `instance.setOption(option)` | `instance.update(options)` | `instance.options(options)` |
| 销毁 | `instance.dispose()` | `instance.destroy()` | `instance.destroy()` |
| resize | `instance.resize()` | `instance.reflow()` | `instance.forceFit()` |
| 主题 | `echarts.registerTheme(name, cfg)` | `options.theme` | `theme` 配置 |
| 事件 | `instance.on(event, handler)` | `instance.on(event, handler)` | `instance.on(event, handler)` |
| 实例类型 | `ECharts` | `Chart` | `Chart` |
| 配置 schema | `EChartsOption` | `Options` | `Options` |

**结论**：六项核心操作（init/update/destroy/resize/theme/event）签名高度同构，Adapter 模式可行性高。配置 schema 因库而异，由 transformer + adapter 联合处理。

### 2.3 现有项目图表封装痛点

通过对常见业务封装的观察，重复痛点集中在：

1. **三态管理**：loading / error / empty 三态每个封装都写一遍，且与 ProTable 的三态实现不一致
2. **resize 监听**：手写 `ResizeObserver` 或 window resize 监听，常常忘记销毁导致内存泄漏
3. **主题联动**：图表主题与 Arco theme 切换不同步，需要业务层手动监听
4. **远程数据**：每个项目各自封装 fetch + setState，无防抖/取消/缓存
5. **更新策略**：setOption 的 `notMerge` / `lazyUpdate` 参数每个项目用法不一，导致渲染闪烁或更新不生效
6. **销毁时机**：React 18 StrictMode 下双重挂载导致 echarts 实例泄漏

ProChart 的设计目标是**一次性解决上述六类问题**，并通过 adapter 让解决方案在所有图表库上生效。

## 三、设计原则

### 3.1 Adapter 模式为核心

```
业务代码
    │
    ▼
  ProChart（React 组件）
    │
    ▼
  ChartAdapter（抽象接口）
    │
    ├── EChartsAdapter ──→ echarts
    ├── HighchartsAdapter ──→ highcharts
    ├── G2Adapter ──→ @antv/g2
    └── 自定义 Adapter ──→ 任意图表库
```

**Adapter 接口必须实现**：`init / update / resize / destroy / getInstance / on / off / setTheme`

**Adapter 不负责**：数据获取、状态管理、三态渲染、Schema 转换（这些由 ProChart 层处理）

### 3.2 Schema 与 Option 双形态，可组合

| 形态 | 适用场景 | 入参 | 心智 |
| ---- | -------- | ---- | ---- |
| **Schema 模式** | 标准 line/bar/pie/scatter 等常见图表 | `type` + `dataSource` + `xField` + `yField` + `seriesField` | 低 |
| **Option 模式** | 需要完整图表库原生配置能力 | `option`（直接传 echarts option 等） | 高 |
| **组合模式** | Schema 模式 + 原生 option 覆盖 | 同时传 `type` + `option`（option 作为 override 深合并） | 中 |

**设计原则**：Schema 模式覆盖 80% 场景，Option 模式作为逃生舱覆盖 20% 复杂场景，二者不互斥。

### 3.3 注册表模式（与 ProForm `componentRegistry` 一致）

```ts
// 全局注册
registerChartAdapter('echarts', EChartsAdapter);
registerChartAdapter('highcharts', HighchartsAdapter);

// 使用
<ProChart adapter="echarts" type="line" dataSource={data} />
// 或直接传 adapter 实例（绕过注册表）
<ProChart adapter={EChartsAdapter} type="line" dataSource={data} />
```

**注册表优势**：
- 支持运行时注册，便于按需 import（`await import('echarts')` 后再注册）
- 支持别名（`'echarts'` / `'ECharts'` 都能命中）
- 与 `componentRegistry` / `instanceRegistry` 模式一致，降低学习成本

### 3.4 按需加载，零默认依赖

- ProChart **不直接依赖** echarts / highcharts / g2，全部为 peer optional
- 内置 adapter 通过独立子路径导出：`@lania-pro-components/components/ProChart/adapters/echarts`
- 不导入任何 adapter 时，ProChart 包体积仅含 React 组件层 + 注册表 + 三态 UI（< 10KB gzipped）
- adapter 支持异步注册：`registerChartAdapter('echarts', () => import('./echarts-adapter'))`

### 3.5 公开 API 默认零破坏

本方案为**新增组件**，无历史 API 负担，但需承诺：

1. 一旦 v1.0 发布，`ProChartProps` / `ChartAdapter` 接口 / `chartAdapterRegistry` API 默认不再做破坏性变更
2. 新增图表类型 / adapter 通过扩展点（transformer 注册表 / adapter 注册表）支持，不修改核心接口
3. 内部 hook（`useChartInstance` 等）允许迭代，但 shared 包导出的类型保持稳定

### 3.6 组合优于配置（与重构方案 P0-3 一致）

- ProChart **不内置**性能监控浮窗（与 ProForm 解耦 PerformanceMonitor 的方向一致）
- 性能打点（`mark/measure`）由 ProChart 内部产生，监控 UI 由使用方通过 `<PerformanceMonitor>` 组合接入
- 三态 UI、工具栏等可选能力通过 children / render prop 注入，不通过 props 配置驱动

### 4.3 Context 架构

与 ProTable 多层 Context 风格保持一致：

```
RootContext（全局配置层）
  ├── adapter（ChartAdapter 实例/名称）
  ├── theme（当前主题，从 useTheme 注入）
  └── performance（性能配置）

DataContext（数据状态层）
  ├── dataSource（原始数据）
  ├── loading / error
  ├── request（远程请求函数）
  └── params（请求参数）

ChartContext（图表实例层）
  ├── instance（adapter 包装的图表实例）
  ├── option（当前生效的 option）
  ├── resize()（手动触发 resize）

```

### 4.4 Adapter 接口设计

```ts
// packages/components/ProChart/adapters/types.ts

/**
 * 图表实例的抽象接口
 * 由具体 adapter 实现，ProChart 仅依赖此接口
 */
export interface ChartInstance<TOption = unknown, TRawInstance = unknown> {
  /** 原始图表实例（escape hatch） */
  readonly raw: TRawInstance;
  /** 更新配置 */
  update(option: TOption, opts?: { notMerge?: boolean; lazyUpdate?: boolean }): void;
  /** 调整尺寸 */
  resize(opts?: { width?: number; height?: number }): void;
  /** 销毁实例 */
  destroy(): void;
  /** 导出图片（base64） */
  toDataURL(type?: 'png' | 'jpeg', bgColor?: string): string;
  /** 转为图片 URL（用于下载） */
  toBlobURL(type?: 'png' | 'jpeg'): Promise<string>;
}

/**
 * 图表适配器接口
 * 不同图表库实现此接口，完成实例生命周期管理
 */
export interface ChartAdapter<TOption = unknown, TRawInstance = unknown> {
  /** adapter 唯一标识，用于注册表查找 */
  readonly name: string;
  /** 创建实例 */
  init(container: HTMLElement, option: TOption, theme?: string): ChartInstance<TOption, TRawInstance>;
  /** 注册事件（返回取消监听函数） */
  on?(instance: ChartInstance<TOption, TRawInstance>, event: string, handler: (payload: unknown) => void): () => void;
  /** 切换主题 */
  setTheme?(instance: ChartInstance<TOption, TRawInstance>, theme: string): void;
  /** 当前主题的默认 option 模板（与用户 option 深合并） */
  getDefaultOption?(theme: string): Partial<TOption>;
}

/**
 * adapter 工厂函数（支持异步，便于动态 import）
 */
export type ChartAdapterFactory<TOption = unknown, TRawInstance = unknown> =
  | ChartAdapter<TOption, TRawInstance>
  | (() => Promise<ChartAdapter<TOption, TRawInstance>>);
```

### 4.5 注册表设计

```ts
// packages/components/ProChart/chartAdapterRegistry.ts

import type { ChartAdapter, ChartAdapterFactory } from './adapters/types';

const registry = new Map<string, ChartAdapterFactory>();
const aliasMap = new Map<string, string>(); // 大小写/别名归一

export function registerChartAdapter(
  name: string,
  factory: ChartAdapterFactory,
  aliases?: string[],
): void {
  const key = name.toLowerCase();
  registry.set(key, factory);
  aliasMap.set(key, key);
  aliases?.forEach((alias) => aliasMap.set(alias.toLowerCase(), key));
}

export function hasChartAdapter(name: string): boolean {
  return aliasMap.has(name.toLowerCase());
}

export async function resolveChartAdapter(
  name: string,
): Promise<ChartAdapter | undefined> {
  const key = aliasMap.get(name.toLowerCase());
  if (!key) return undefined;
  const factory = registry.get(key);
  if (!factory) return undefined;
  return typeof factory === 'function' ? await factory() : factory;
}

export function unregisterChartAdapter(name: string): void {
  const key = aliasMap.get(name.toLowerCase());
  if (!key) return;
  registry.delete(key);
  for (const [alias, target] of aliasMap) {
    if (target === key) aliasMap.delete(alias);
  }
}
```

## 五、与架构公共化重构方案的关联

ProChart 的落地**强依赖** [架构公共化重构方案.md](../架构公共化重构方案.md) 的多个任务。本节明确依赖关系，并指出 ProChart 反向贡献给 shared 包的新增内容。

### 5.1 ProChart 对重构方案的依赖

| ProChart 任务 | 依赖的重构方案任务 | 依赖原因 | 阻塞性 |
| ------------ | ----------------- | -------- | ------ |
| PC-0 脚手架 | **M0 shared 包脚手架** | `useChartInstance` 等需沉淀到 shared 包 | 强阻塞 |
| PC-1 核心 | **P1-1 useAsyncRequest** | 远程数据加载委托给 useAsyncRequest | 弱阻塞（可先用 ProTable RequestEngine 临时方案） |
| PC-1 核心 | **P1-4 useCache / useResponsive** | 数据缓存 + 容器 resize | 弱阻塞 |
| PC-3 性能打点 | **P0-3 PerformanceMonitor** | `mark('chart-render')` / `measure` 数据源 | 弱阻塞（可先用 utils 的 performanceMonitor 单例） |
| PC-6 主题 | theme 包（已存在） | light/dark → chart theme 映射 | 无阻塞 |
| PC-2 Schema 模式 | utils/format、utils/object、utils/defineEnumMap | tooltip 格式化 + option 合并 + 枚举映射 | 无阻塞 |

**结论**：ProChart 可以与重构方案 P0/P1 任务**并行启动**，仅 M0 shared 脚手架为强阻塞。建议在 M0 完成后立即启动 PC-0，与 P0/P1 任务并行推进。

### 5.2 ProChart 对 shared 包的反向贡献

ProChart 落地过程中产生两个**通用 hook**，应上提到 shared 包，作为对重构方案 P1-4 的补充：

| 新增 hook | 位置 | 用途 | 潜在复用方 |
| --------- | ---- | ---- | ---------- |
| `useChartInstance` | `shared/hooks/useChartInstance.ts` | 命令式实例生命周期管理（init/update/destroy 时机，StrictMode 安全） | ProTable 可编辑表格、ProDialog 命令式实例 |
| `useResizeObserver` | `shared/hooks/useResizeObserver.ts` | ResizeObserver 封装，自动销毁 | ProTable 虚拟滚动容器、ProDialog 弹窗内容自适应 |

**建议**：在重构方案 P1-4 任务清单中补充 `useChartInstance` / `useResizeObserver` 上提项，作为 P1-5（或合并入 P1-4）。

### 5.3 ProChart 与架构债务的协同

| 重构方案债务 | ProChart 落地时是否触及 | 协同方式 |
| ----------- | ---------------------- | -------- |
| #3 RequestEngine AbortController 未透传 | 间接 | ProChart 消费 useAsyncRequest 时若发现透传缺陷，可顺带修复（与 P1-1 合并 PR） |
| #8 ProForm performance.monitor 配置耦合 | 无 | 不涉及 |
| #1 window.dispatchEvent 全局广播 | 无 | ProChart 不使用全局事件 |

### 5.4 在重构方案任务清单中的位置

建议在重构方案文档的**任务清单**中新增 P3 级别任务：

**line transformer 示例（echarts option 输出）**：

```ts
// packages/components/ProChart/transformers/line.ts

import { registerChartTransformer } from './types';
import type { ChartSchema } from '../ChartSchema';

registerChartTransformer({
  type: 'line',
  transform(schema: ChartSchema, ctx: TransformContext) {
    const { dataSource = [], xField, yField, seriesField, series } = schema;
    if (!xField || !yField) {
      throw new Error('[ProChart] line transformer requires xField and yField');
    }

    const yFields = Array.isArray(yField) ? yField : [yField];

    // 单系列无分组
    if (!seriesField) {
      return {
        xAxis: { type: 'category', data: dataSource.map((d) => d[xField]) },
        yAxis: { type: 'value' },
        series: yFields.map((yf) => ({
          type: 'line',
          name: yf,
          smooth: series?.smooth,
          areaStyle: series?.area ? {} : undefined,
          data: dataSource.map((d) => d[yf]),
        })),
      };
    }

    // 多系列（按 seriesField 分组）
    const groups = new Map<string, Record<string, unknown>[]>();
    dataSource.forEach((d) => {
      const key = String(d[seriesField!]);
      if (!groups.has(key)) groups.set(key, []);
      groups.get(key)!.push(d);
    });

    const xData = Array.from(
      new Set(dataSource.map((d) => d[xField!])),
    );

    return {
      xAxis: { type: 'category', data: xData },
      yAxis: { type: 'value' },
      series: Array.from(groups.entries()).map(([name, rows]) => ({
        type: 'line',
        name,
        smooth: series?.smooth,
        areaStyle: series?.area ? {} : undefined,
        data: xData.map((x) => {
          const row = rows.find((r) => r[xField!] === x);
          return row ? row[yFields[0]] : null;
        }),
      })),
    };
  },
});
```

### 7.4 EChartsAdapter 实现要点

```ts
// packages/components/ProChart/adapters/echarts/EChartsAdapter.ts（要点节选）

import * as echarts from 'echarts'; // 由消费方按需引入，adapter 不打包
import type { ChartAdapter, ChartInstance } from '../types';
import { lightTheme, darkTheme } from './echartsTheme';

export interface EChartsOption { /* 等同 echarts EChartsOption */ }

export const EChartsAdapter: ChartAdapter<EChartsOption, echarts.ECharts> = {
  name: 'echarts',

  init(container, option, theme = 'light') {
    const themeCfg = theme === 'dark' ? darkTheme : lightTheme;
    const raw = echarts.init(container, themeCfg);
    raw.setOption(option);
    return {
      raw,
      update: (opt, o) => raw.setOption(opt, { notMerge: o?.notMerge ?? false, lazyUpdate: o?.lazyUpdate ?? false }),
      resize: (o) => raw.resize(o ?? {}),
      destroy: () => raw.dispose(),
      toDataURL: (type = 'png', bgColor) =>
        raw.getDataURL({ type, backgroundColor: bgColor, pixelRatio: 2 }),
      toBlobURL: async (type = 'png') => {
        const dataURL = raw.getDataURL({ type, pixelRatio: 2 });
        const blob = await (await fetch(dataURL)).blob();
        return URL.createObjectURL(blob);
      },
    };
  },

  on(instance, event, handler) {
    instance.raw.on(event, handler);
    return () => instance.raw.off(event, handler);
  },

  setTheme(instance, theme) {
    // echarts 切换主题需要重新 init（限制）
    // 方案：保存当前 option，dispose 后用新主题重新 init
    const currentOption = instance.raw.getOption();
    const container = instance.raw.getDom();
    instance.raw.dispose();
    const themeCfg = theme === 'dark' ? darkTheme : lightTheme;
    const newRaw = echarts.init(container, themeCfg);
    newRaw.setOption(currentOption);
    // 注意：调用方需替换 instance 引用
  },

  getDefaultOption(theme) {
    return {
      grid: { left: 40, right: 20, top: 40, bottom: 40, containLabel: true },
      tooltip: { trigger: 'axis' },
      legend: { top: 0 },
      color: theme === 'dark' ? darkTheme.color : lightTheme.color,
    };
  },
};
```

### 7.5 生命周期与 StrictMode 安全

React 18 StrictMode 下 `useEffect` 会执行两次（mount → unmount → mount），echarts 实例若管理不当会泄漏。ProChart 采用以下策略：

```tsx
// ProChart.tsx 核心生命周期管理（要点节选）

const containerRef = useRef<HTMLDivElement>(null);
const instanceRef = useRef<ChartInstance | null>(null);
const destroyedRef = useRef(false);

useEffect(() => {
  if (!containerRef.current || !adapter || !option) return;
  destroyedRef.current = false;

  performanceMonitor.mark('chart-render-start');

  // 异步 resolve adapter（支持动态 import）
  let cancelled = false;
  resolveChartAdapter(typeof adapter === 'string' ? adapter : adapter.name)
    .then((resolvedAdapter) => {
      if (cancelled || destroyedRef.current || !resolvedAdapter) return;

      const inst = resolvedAdapter.init(
        containerRef.current!,
        option,
        resolvedTheme,
      );
      instanceRef.current = inst;
      onChartReady?.(inst);

      performanceMonitor.measure('chart-render', 'chart-render-start');
    });

  return () => {
    cancelled = true;
    destroyedRef.current = true;
    // StrictMode 第二次 mount 时，旧实例必须销毁
    if (instanceRef.current) {
      instanceRef.current.destroy();
      instanceRef.current = null;
    }
  };
}, [adapter]); // 注意：option 变化走 update，不重新 init
```

## 十、风险与回滚

### 10.1 风险等级

| 任务 | 风险 | 主要风险点 | 缓解措施 |
| ---- | ---- | ---------- | -------- |
| MC-0 | 低 | 仅脚手架 | — |
| MC-1 | 中 | StrictMode 双重挂载、adapter 异步 resolve 时序 | 充分测试 + cancelled flag |
| MC-2 | 中 | transformer 适配三种图表库差异 | 首期仅保证 echarts，highcharts/g2 在 MC-3 处理 |
| MC-3 | 中 | 三种图表库 option 差异大 | transformer 内分支处理 + 视觉等价测试 |
| MC-4 | 低 | 复用已有抽象 | — |
| MC-5 | 低 | 纯迁移 | — |
| MC-6 | 低 | 主题映射表设计 | 参考 Arco token |

### 10.2 已知设计限制

1. **echarts 主题切换需重新 init**：echarts 不支持运行时切换主题，`setTheme` 实现需要 dispose + 重新 init。可能造成短暂闪烁。
   - **缓解**：在 adapter 中保存当前 option，重新 init 后立即 setOption，最小化闪烁
2. **highcharts 商业授权**：highcharts 商用需购买授权，文档需显著提示
3. **transformer 跨库等价性**：三种图表库的默认行为差异（动画、tooltip 样式、颜色循环）无法 100% 一致
   - **缓解**：文档明确"视觉等价"而非"像素级一致"

### 10.3 回滚策略

- ProChart 为**新增组件**，回滚仅需删除目录 + 移除导出，对现有组件零影响
- 已注册的 adapter 可通过 `unregisterChartAdapter` 清理
- shared 包新增的 `useChartInstance` / `useResizeObserver` 即使 ProChart 回滚，仍可被其他组件复用，无需回滚

### 10.4 灰度策略

- MC-1 完成后在内部项目试用 1-2 周，验证 StrictMode / 主题 / resize 行为
- MC-3（多 adapter）完成后再次灰度，验证切换场景
- 正式发布前完成 docs 示例 + README

### 11.2 术语表

| 术语 | 含义 |
| ---- | ---- |
| Adapter | 图表库适配器，抽象 init/update/resize/destroy 等生命周期 |
| Transformer | Schema → option 的转换器，按图表类型实现 |
| Schema 模式 | 通过 `type + dataSource + xField/yField` 声明图表，低门槛 |
| Option 模式 | 直接传图表库原生 option，高灵活 |
| chartAdapterRegistry | adapter 运行时注册表，支持异步 factory |
| 视觉等价 | 同 Schema + 不同 adapter 渲染结果视觉一致，非像素级一致 |

### 11.3 决策记录

| 决策点 | 选择 | 理由 | 日期 |
| ------ | ---- | ---- | ---- |
| 抽象模式 | Adapter 模式 | 图表库生命周期高度同构，adapter 可行性高；与 ProForm componentRegistry 模式一致 | 2026-07-04 |
| Adapter 注册 | 注册表 + 异步 factory | 支持动态 import，保证 Tree Shaking | 2026-07-04 |
| 双形态 | Schema + Option 组合 | Schema 覆盖 80% 场景，Option 作为逃生舱 | 2026-07-04 |
| 默认 adapter | 无默认（必须显式指定） | 避免隐式引入大体积依赖 | 2026-07-04 |
| 内置 adapter 数量 | 3 种（echarts/highcharts/g2） | 覆盖主流场景，其他通过自定义 adapter 支持 | 2026-07-04 |
| 性能监控 | 组合式（与 P0-3 一致） | ProChart 不内置监控 UI，仅产生打点数据 | 2026-07-04 |
| 主题切换 | adapter.setTheme（echars 需重新 init） | 接受 echarts 限制，最小化闪烁 | 2026-07-04 |
| hooks 上提 | useChartInstance / useResizeObserver 上提到 shared | 通用 hook，可被其他组件复用 | 2026-07-04 |
