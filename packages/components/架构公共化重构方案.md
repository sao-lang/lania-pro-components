## 一、背景与目标

### 1.1 背景

`@lania-pro-components/components` 包含 6 个组件：ActionButton、ProDialog、ProForm、ProTable、ProSelect、ProUpload。经过多轮迭代，组件能力日趋完善，但也积累了**跨组件的架构重复**和**抽象边界不清**两类问题：

- 同一能力（虚拟滚动、实例注册表、远程请求）在多个组件中**独立实现**
- 同一模式（Provider hook、Context 分层、命令式 createRoot）**风格不统一**
- 部分通用配置类型（性能/缓存/响应式）**散落在单组件内**
- 部分架构债务（全局事件广播、Provider 快照过期、AbortController 未透传）**未及时修复**

底层工具已成功下沉到 `@lania-pro-components/utils`（响应式系统、性能工具、格式化函数），但**组件层缺少对应的公共抽象层**。

### 1.2 目标

1. **消除明确的重复实现**：把多处独立实现的同一能力合并为单一来源
2. **建立组件层公共抽象**：在新建的 `packages/shared/` 包内沉淀跨组件复用的组件、Hook、类型、工厂
3. **统一架构模式**：Provider 模式、Registry 模式、Store 订阅模式、命令式渲染模式形成约定
4. **顺带修复架构债务**：在不破坏 API 的前提下修掉已发现的 bug
5. **保持公开 API 稳定**：所有重构对消费方零破坏，仅内部结构变化

### 1.3 范围

**包含**：

- `packages/components/` 下 6 个组件的内部架构重构
- `packages/utils/` 新增纯函数/类
- 新建 `packages/shared/` 独立 workspace 包（React 抽象底座）

**不包含**：

- `packages/theme/` 改造（无可抽函数）
- 公开 API 的破坏性变更
- 新业务能力开发

## 三、设计原则

### 3.1 公开 API 默认零破坏，例外需显式标注

所有重构**默认**必须保持消费方导入路径、组件 props、Ref 方法签名、Hook 返回值结构**完全不变**。内部实现可重写，外部契约不动。

**例外**：当某项重构的核心目标就是"改变外部契约"时（如 P0-3 的 PerformanceMonitor 解耦，必须删除 `performance.monitor` 配置才能达成组合优于配置的设计目标），允许破坏性变更，但必须满足：

1. 在任务描述中明确标注"破坏性变更"
2. 提供完整的迁移指南（旧用法 → 新用法对照）
3. 在 README / CHANGELOG 中显著说明
4. 主版本号 bump（minor 或 major）

**校验方式**：每个 PR 必须通过 `tsc` 类型检查 + 现有测试套件 + docs 示例编译。破坏性变更的 PR 还需更新所有受影响的 docs 示例。

### 3.2 公共化边界判定

一个抽象是否应该公共化，按以下决策树判断：

```
该抽象是否纯函数/类（无 React/Arco 依赖）？
├─ 是 → 迁移到 packages/utils/
└─ 否 → 该抽象是否被 2 个以上组件需要？
    ├─ 是 → 迁移到 packages/shared/
    └─ 否 → 保留在单组件内
```

**反模式禁止**：

- ❌ 为了"看起来更通用"而过度抽象（YAGNI）
- ❌ 把单组件特有的语义强抽到公共层（如把 `ProFormSchema` 抽到 shared）
- ❌ 在 utils 中引入 React/Arco 依赖（破坏 utils 零依赖原则）

### 3.3 命名约定

| 层           | 命名规则                                       | 示例                                            |
| ------------ | ---------------------------------------------- | ----------------------------------------------- |
| utils 纯函数 | `camelCase`                                    | `createInstanceRegistry`、`getImageSize`        |
| utils 类     | `PascalCase`                                   | `InstanceRegistry`、`DraftEngine`               |
| shared hooks | `useXxx`                                       | `useVirtualScroll`、`useAsyncRequest`           |
| shared 组件  | `PascalCase`                                   | `PerformanceMonitor`                            |
| shared 类型  | `PascalCase` + `Config`/`Props`/`Options` 后缀 | `PerformanceConfig`、`AsyncDataConfig`          |
| shared 工厂  | `createXxx`                                    | `createProProvider`、`createImperativeInstance` |

### 3.4 渐进式迁移

每个 P0/P1 任务**独立成 PR**，不合并提交。每个 PR 必须：

1. 引入新的公共抽象
2. 迁移 1 个消费方作为示范
3. 保留旧实现（标记 `@deprecated`）让其他消费方渐进迁移
4. 全部消费方迁移完成后，单独 PR 删除旧实现

### 3.5 测试先行

每个公共抽象必须有单元测试，覆盖：

- 基本功能
- 边界情况
- 与原实现的等价性（回归测试）

## 四、分层方案选型

### 4.1 方案对比

| 方案                                        | 描述                                             | 优点                                                                 | 缺点                                                                   |
| ------------------------------------------- | ------------------------------------------------ | -------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| **A. 保守：`packages/components/_shared/`** | 在 components 包内加内部共享目录，不暴露为子路径 | 改动小（无需改 rollup/tsconfig/package.json 的子路径配置）；快速落地 | 外部消费者无法直接 import 公共组件                                     |
| **B. 激进：新建 `packages/shared/` 包**     | 独立 workspace 包，独立构建发布                  | 边界清晰，可独立发布，外部可用                                       | 改动大（rollup/tsconfig/vitest/package.json 全要改）；引入新包管理成本 |

### 4.2 选定方案：B（激进独立包）

**理由**：

1. **边界清晰**：独立包强制开发者思考"这个抽象是否真的跨组件通用"，避免 `_shared/` 沦为杂物间
2. **可独立发布**：外部消费者可直接 `import { PerformanceMonitor } from '@lania-pro-components/shared'`，不必依赖整个 components 包
3. **与 utils 对称**：utils 是"纯函数底座"（零 React 依赖），shared 是"React 抽象底座"（React 感知但 Arco 解耦），形成清晰的双层公共底座
4. **发布独立**：shared 可独立版本号、独立 changelog，components 升级 shared 不影响 utils
5. **未来扩展性**：shared 可承载更多 React 通用能力（虚拟列表、命令式 API 框架、Provider 工厂），不受 components 包大小约束

**代价接受度**：构建配置改动是一次性成本，后续维护成本反而更低（独立包边界清晰）。

## 五、任务清单（按优先级）

### P0 任务（必须做，低风险高收益）

#### **P0-1：合并 instanceRegistry → utils**

**动机（Why）**

[ProDialog/instanceRegistry.ts:34](./ProDialog/instanceRegistry.ts) 与 [ProForm/registry/instanceRegistry.ts:23](./ProForm/registry/instanceRegistry.ts) 是两份几乎同构的实现：

- ProDialog 版：`InstanceRegistry` 类，方法 `register/unregister/get/getAll/clear/has`，全局单例
- ProForm 版：`ProFormInstanceRegistry` 类，方法 `register/unregister/get/getAllNames/clear/has` + `subscribe` 订阅机制

ProForm 自己的注释里就写明"与 ProDialog 的 InstanceRegistry 设计一致"。两份代码各自维护，bug 修一份漏一份。

**改动范围（What）**

1. **新增** `packages/utils/src/instanceRegistry.ts`：
   ```ts
   export class InstanceRegistry<T> {
     register(name: string, instance: T): void;
     unregister(name: string): void;
     get(name: string): T | undefined;
     has(name: string): boolean;
     getAll(): Map<string, T>;
     clear(): void;
     subscribe(listener: (name: string, instance: T | undefined) => void): () => void; // 新增
   }
   export function createInstanceRegistry<T>(): InstanceRegistry<T>;
   ```

**影响**：

### 10.2 术语表

| 术语           | 含义                                                                   |
| -------------- | ---------------------------------------------------------------------- |
| 公共化         | 把散落或重复的抽象提取到共享位置                                       |
| utils 包       | `@lania-pro-components/utils`，纯函数/类，零 React/Arco 依赖           |
| shared 包      | `@lania-pro-components/shared`，React 抽象底座，React 感知但 Arco 解耦 |
| 公开 API       | 消费方可见的接口（props / Ref / Hook 返回值 / 导出名）                 |
| 内部 API       | 组件内部使用的接口，不对消费方承诺稳定性                               |
| @deprecated 壳 | 保留旧导出名但内部委托新实现的过渡文件                                 |
| 组合优于配置   | 监控/调试等运维 UI 由使用方组合接入，不通过组件 prop 配置驱动          |
