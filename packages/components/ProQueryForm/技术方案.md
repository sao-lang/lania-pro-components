## 一、背景与目标

### 1.1 背景

`@lania-pro-components/components` 当前的查询表单能力**内嵌于 ProTable**（[ProTable/features/QueryForm.tsx](../ProTable/features/QueryForm.tsx)），未独立成组件。这导致：

- **复用受限**：详情页、图表页等非表格场景无法独立使用查询表单
- **ProTable 耦合**：QueryForm 通过三层 Context（DataContext/ColumnContext/RootContext）深度依赖 ProTable，无法脱离 ProTable 使用
- **与 ProTable 列定义强绑定**：查询字段只能从 `columns` 派生，无法独立声明 `schemas`
- **架构债务**：[QueryForm.tsx](../ProTable/features/QueryForm.tsx) 在早返回之后调用 `useMemo`/`useEffect`（第 251、272 行），违反 React Hooks 规则
- **能力分散**：URL 同步（useUrlSync）、查询方案管理（useSearchSchema）等通用能力散落在 ProTable/hooks，无法被非 ProTable 场景复用

### 1.2 目标

1. **独立组件**：ProQueryForm 作为独立组件，脱离 ProTable 使用，支持非表格场景（图表查询、详情筛选）
2. **双形态入参**：支持 `columns`（ProTable 列驱动，内部转 schema）和 `schemas`（纯 ProForm Schema 驱动）两种入参形态
3. **复用 ProForm 引擎**：直接 `import { ProForm }` 渲染表单，复用 ProFormSchema / ProFormInstance / SchemaProcessOptions 类型
4. **复用通用 hooks**：消费 `useUrlSync` / `useSearchSchema`（P2-1 上提到 shared），与 ProTable 共用同一套 URL 同步和查询方案逻辑
5. **轻量优先**：默认只管表单状态，通过 `onSearch` / `onReset` 回调与上层解耦；可选 `store` prop 集成 DataStore 实现重量模式
6. **修复 React Hooks bug**：独立化过程中修正 [QueryForm.tsx](../ProTable/features/QueryForm.tsx) 的 hooks 顺序问题

### 1.3 范围

**包含**：

- `packages/components/ProQueryForm/` 新增组件目录
- 从 ProTable 迁移 4 个纯函数（`valueTypeToComponent` / `getComponentPropsByValueType` / `convertColumnsToSearchSchema` / `transformSearchParams`）
- ProTable QueryForm 改为 re-export ProQueryForm（向后兼容）

**不包含**：

- 不重新实现 ProForm Schema 引擎（直接复用）
- 不重新实现 useUrlSync / useSearchSchema（直接复用，P2-1 上提后从 shared import）
- 不修改 ProTable 的公开 API（QueryForm 改为 re-export 壳，行为不变）

## 二、现状与动机

### 2.1 现有可复用底座扫描

| 复用对象 | 来源 | ProQueryForm 用途 |
| -------- | ---- | ----------------- |
| `ProForm` | [ProForm/index.ts](../ProForm/index.ts) | 渲染查询表单（直接 import 组件） |
| `ProFormSchema` / `ProFormInstance` / `SchemaProcessOptions` | [ProForm/types.ts](../ProForm/types.ts) | 字段定义类型、表单实例类型、Schema 自动补全配置 |
| `ProColumnType.search` | [ProTable/types.ts:315](../ProTable/types.ts) | 列级搜索配置（已是 `Omit<ProFormSchema,'name'> + {order?, transform?}`） |
| `SearchConfig` | [ProTable/types.ts:1044](../ProTable/types.ts) | ProTable 的 search 配置（layout/columns/collapsible/formProps/beforeSearch 等） |
| `useUrlSync` | [ProTable/hooks/useUrlSync.ts](../ProTable/hooks/useUrlSync.ts)（P2-1 上提到 shared） | URL 双向同步，仅依赖 DataStoreImpl |
| `useSearchSchema` | [ProTable/hooks/useSearchSchema.ts](../ProTable/hooks/useSearchSchema.ts)（P2-1 上提到 shared） | 查询方案 CRUD + localStorage 持久化，零 ProTable 依赖 |
| `SearchSchemaSelector` | [ProTable/components/SearchSchemaSelector.tsx](../ProTable/components/SearchSchemaSelector.tsx) | 查询方案选择 UI（纯组件，仅依赖 SearchSchema 类型） |
| `DataStoreImpl` | [ProTable/store/DataStore.ts](../ProTable/store/DataStore.ts) | 重量模式集成（可选） |
| `convertColumnsToSearchSchema` 等 4 个纯函数 | [ProTable/features/QueryForm.tsx](../ProTable/features/QueryForm.tsx) | 列驱动模式的 schema 转换（需迁移） |

**结论**：90% 的能力已存在，ProQueryForm 是上述能力的"独立化封装"，核心工作是迁移 + 解耦 + 修 bug。

### 2.2 现有 ProTable QueryForm 的架构债务

| 问题 | 位置 | 影响 | ProQueryForm 修正方式 |
| ---- | ---- | ---- | --------------------- |
| **React Hooks 顺序违规** | [QueryForm.tsx:251,272](../ProTable/features/QueryForm.tsx) | 在 `if (!search) return null;` 和 `if (searchSchemas.length === 0) return null;` 早返回之后调用 `useMemo`/`useEffect` | 所有 hooks 提到早返回之前 |
| **三层 Context 强耦合** | [QueryForm.tsx](../ProTable/features/QueryForm.tsx) 全文 | 依赖 DataContext/ColumnContext/RootContext，无法脱离 ProTable | ProQueryForm 通过 props 接收所有配置，不依赖 Context |
| **仅支持 columns 入参** | [QueryForm.tsx](../ProTable/features/QueryForm.tsx) `convertColumnsToSearchSchema` | 无法独立声明 schemas（非表格场景无法用） | ProQueryForm 支持 `columns` 和 `schemas` 双形态入参 |
| **formRef 单一共享** | [ProTable/index.tsx:126](../ProTable/index.tsx) | formRef 同时给 QueryForm/SearchSchemaSelector/action 用，耦合严重 | ProQueryForm 通过 forwardRef 暴露 ProFormInstance，或通过 useProFormContext |

### 2.3 4 个纯函数的迁移价值

[QueryForm.tsx](../ProTable/features/QueryForm.tsx) 内的 4 个纯函数是 ProTable→ProForm 的 schema 桥梁，迁移到 ProQueryForm 后：

| 函数 | 签名 | 迁移价值 |
| ---- | ---- | -------- |
| `valueTypeToComponent` | `Record<Exclude<ProColumnValueType,'opr'\|'proTable'>, string>` | valueType → Arco 组件名映射（text→Input / number→InputNumber / select→Select / date→DatePicker 等） |
| `getComponentPropsByValueType` | `(valueType, column) => Record<string, unknown>` | 按 valueType 生成 width/format/options/precision/placeholder |
| `convertColumnsToSearchSchema` | `<T>(columns: ProColumnType<T>[]) => ProFormSchema[]` | 列驱动模式核心：过滤 hideInSearch/search===false/valueType==='opr'/无 dataIndex；支持 search.order 排序 |
| `transformSearchParams` | `(params, columns) => Record<string, unknown>` | 查询参数转换：删空值 + 逐列调用 search.transform |

**迁移策略**：迁移到 `ProQueryForm/utils.ts`，ProTable/features/QueryForm.tsx 改为 re-export（保持向后兼容）。

### 2.4 与 ProTable 的解耦难点

| 难点 | 说明 | 解决方案 |
| ---- | ---- | -------- |
| `useUrlSync` 依赖 DataStoreImpl | store-centric 设计，ProQueryForm 轻量模式无 store | 轻量模式不启用 useUrlSync；重量模式（传 store prop）才启用 |
| `useRequest` 强耦合分页/排序 | ProQueryForm 仅管查询参数，不涉及分页 | ProQueryForm 不复用 useRequest，通过 onSearch 回调由消费方触发请求 |
| `setQuery` 副作用（重置 page=1 + 清空选择） | DataStore 设计强耦合表格语义 | 轻量模式无此副作用；重量模式文档明确说明副作用 |
| `SearchSchemaSelector` 依赖 formRef | ProTable 中 formRef 共享给多个组件 | ProQueryForm 通过 useProFormContext 或 forwardRef 暴露 ProFormInstance |

**结论**：通过"轻量模式 + 重量模式"双形态设计，ProQueryForm 既能独立使用，又能与 ProTable/DataStore 集成。

## 三、设计原则

### 3.1 双形态入参为核心

```
业务代码
    │
    ▼
  ProQueryForm（React 组件）
    │
    ├── 形态 A：columns 列驱动
    │   └── convertColumnsToSearchSchema → ProFormSchema[]
    │
    └── 形态 B：schemas Schema 驱动
        └── 直接使用 ProFormSchema[]
    │
    ▼
  ProForm（渲染表单）
    │
    ▼
  onSearch(params) / onReset()
    │
    ├── 轻量模式：消费方自行处理（fetch / setState）
    └── 重量模式：ProQueryForm 内部 store.setQuery(params) + store.reload()
```

**设计原则**：形态 A 覆盖 ProTable 列定义互通场景，形态 B 覆盖独立查询场景；轻量模式覆盖 80% 场景，重量模式覆盖与 ProTable 集成的 20% 场景。

### 3.2 双模式：轻量 + 重量

| 模式 | 触发条件 | 行为 | 适用场景 |
| ---- | -------- | ---- | -------- |
| **轻量模式**（默认） | 未传 `store` prop | 仅管理表单状态，通过 `onSearch`/`onReset` 回调通知消费方 | 图表查询、详情筛选、自定义列表 |
| **重量模式** | 传 `store` prop（DataStoreImpl 实例） | 内部调用 `store.setQuery(params)`，自动触发 store 订阅者（如 ProTable useRequest） | 与 ProTable 深度集成，替代内嵌 QueryForm |

### 3.3 复用优于重新实现

- 表单渲染：**直接 import** `ProForm` 组件
- 类型定义：**直接 import** `ProFormSchema` / `ProFormInstance` / `SchemaProcessOptions`
- URL 同步：**直接 import** `useUrlSync`（P2-1 上提后从 shared）
- 查询方案：**直接 import** `useSearchSchema`（P2-1 上提后从 shared）
- 方案选择 UI：**直接 import** `SearchSchemaSelector`
- schema 转换：**迁移** 4 个纯函数到 `ProQueryForm/utils.ts`

### 3.4 公开 API 默认零破坏

本方案为**新增组件**，但需承诺 ProTable 向后兼容：

1. ProTable 的 `search` 配置行为**完全不变**（内部改为 re-export ProQueryForm）
2. ProTable 的 `QueryForm` 导出**完全不变**（改为 re-export ProQueryForm 的壳）
3. ProQueryForm 自身的 `ProQueryFormProps` / `ProQueryFormInstance` 类型默认不再做破坏性变更
4. 4 个纯函数迁移后，ProTable/features/QueryForm.tsx 改为 re-export，保持导入路径兼容

### 3.5 组合优于配置（与重构方案 P0-3 一致）

- ProQueryForm **不内置**性能监控浮窗
- 性能打点（`mark/measure`）由 ProQueryForm 内部产生（`query-form-render`），监控 UI 由使用方通过 `<PerformanceMonitor>` 组合接入
- 查询方案管理 UI（SearchSchemaSelector）通过 props 注入，不强制启用

## 四、架构设计

### 4.1 模块划分

```
ProQueryForm
├── Core Layer（核心层）
│   ├── ProQueryForm.tsx              # 主组件
│   ├── ProQueryFormContext.tsx       # 上下文（与 ProTable 多层 Context 风格一致）
│   └── types.ts                      # ProQueryFormProps / ProQueryFormInstance 类型
│
├── Utils Layer（工具层，从 ProTable 迁移）
│   ├── utils/
│   │   ├── valueTypeToComponent.ts   # valueType → Arco 组件名映射
│   │   ├── columnToProps.ts          # getComponentPropsByValueType
│   │   ├── columnsToSchema.ts        # convertColumnsToSearchSchema
│   │   ├── transformParams.ts        # transformSearchParams
│   │   └── index.ts
│
├── Feature Layer（功能层）
│   ├── QueryFormRenderer.tsx         # 渲染 ProForm + 查询/重置按钮
│   └── SearchSchemaBar.tsx           # 查询方案管理条（包装 SearchSchemaSelector）
│
└── index.ts                          # barrel
```

### 4.2 数据流

```
   ┌──────────────────────┐
   │  columns 或 schemas   │ ← 形态 A：ProColumnType[] / 形态 B：ProFormSchema[]
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  convertColumnsToSchema │ ← 仅形态 A：列驱动转 schema
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  ProForm 渲染          │ ← 直接复用 ProForm
   │  + 查询/重置按钮       │
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  onSearch(values)     │ ← 用户点击查询 / 表单提交
   │  → transformSearchParams │
   │  → beforeSearch?.(params) │
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  轻量模式：onSearch?.(params) │ ← 消费方自行处理
   │  重量模式：store.setQuery(params) │ ← 自动触发 store 订阅者
   └──────────────────────┘
```

### 4.3 Context 架构

与 ProTable 多层 Context 风格保持一致：

```
RootContext（全局配置层）
  ├── layout（表单布局：horizontal/vertical/inline）
  ├── columns（列数配置）
  ├── collapsible / defaultCollapsed / collapsedRows
  └── theme（当前主题，从 useTheme 注入）

SchemaContext（字段定义层）
  ├── schemas（处理后的 ProFormSchema[]）
  └── formRef（ProFormInstance 引用）

ActionContext（行为层）
  ├── onSearch(params)
  ├── onReset()
  └── store?（重量模式的 DataStoreImpl 实例）
```

## 五、与架构公共化重构方案的关联

ProQueryForm 的落地**弱依赖** [架构公共化重构方案.md](../架构公共化重构方案.md) 的部分任务。本节明确依赖关系。

### 5.1 ProQueryForm 对重构方案的依赖

| ProQueryForm 任务 | 依赖的重构方案任务 | 依赖原因 | 阻塞性 |
| ----------------- | ----------------- | -------- | ------ |
| MQ-0 脚手架 | 无 | 仅新增目录 | 无阻塞 |
| MQ-1 核心 | 无 | 直接 import ProForm / ProTable 类型 | 无阻塞 |
| MQ-2 通用 hooks 复用 | **P2-1 useUrlSync / useSearchSchema 上提** | 从 shared import 通用版 | 弱阻塞（可先用 ProTable 版） |
| MQ-3 ProTable 集成 + bug 修复 | 无 | 直接 import ProTable/features/QueryForm 改为 re-export | 无阻塞 |

**结论**：ProQueryForm 可以**立即启动**，MQ-2 弱依赖 P2-1，可先用 ProTable/hooks 内联版本作为过渡，待 P2-1 完成后切换 import 路径。

### 5.2 ProQueryForm 对 shared 包的反向贡献

ProQueryForm 落地过程中**不产生**新的通用 hook，但反向验证了以下抽象的合理性：

| 验证对象 | 验证方式 |
| -------- | -------- |
| `useUrlSync` 上提的必要性 | ProQueryForm 在非 ProTable 场景消费此 hook，验证其是否真的"通用 URL 同步"而非"ProTable URL 同步" |
| `useSearchSchema` 上提的必要性 | ProQueryForm 在非 ProTable 场景消费此 hook，验证其是否真的"通用查询方案管理" |
| `ProFormSchema` 的可复用性 | ProQueryForm 直接消费 ProForm 的 schema 类型，验证其是否真的"通用表单 schema" |
| `ProColumnType.search` 的派生能力 | 通过 `Omit<ProFormSchema,'name'>` 派生，验证列定义与 schema 的互通设计 |

**建议**：若 MQ-2 发现 `useUrlSync` 含 ProTable 语义耦合（如依赖 `store.sorter`），可在 P2-1 任务中顺带解耦。

### 5.3 ProQueryForm 与架构债务的协同

| 重构方案债务 | ProQueryForm 落地时是否触及 | 协同方式 |
| ----------- | --------------------------- | -------- |
| **QueryForm React Hooks 顺序违规**（新发现） | 直接触及 | MQ-3 迁移时修正，与 P3-3 合并 PR |
| #1 window.dispatchEvent 全局广播 | 间接 | DataStore.reload() 派发 `protable:reload` 事件，ProQueryForm 重量模式继承此行为 |

### 5.4 在重构方案任务清单中的位置

ProQueryForm 已纳入重构方案 **P3-3** 任务清单（见 [架构公共化重构方案.md](../架构公共化重构方案.md) P3-3 组件清单表）。本方案为 P3-3 的子文档之一。

### MQ-1：核心渲染 + 轻量模式（MVP）

**依赖**：MQ-0

**改动范围**：

1. **迁移** 4 个纯函数到 `ProQueryForm/utils/`：
   - `valueTypeToComponent.ts`：从 [QueryForm.tsx](../ProTable/features/QueryForm.tsx) 迁移
   - `columnToProps.ts`：`getComponentPropsByValueType`
   - `columnsToSchema.ts`：`convertColumnsToSearchSchema`
   - `transformParams.ts`：`transformSearchParams`
   - `utils/index.ts`：barrel
2. **新增** `ProQueryForm/QueryFormRenderer.tsx`：渲染 ProForm + 查询/重置按钮
   - 接收 `schemas`（已转换）+ `formProps`（透传 ProForm）
   - 调用 `<ProForm schemas={schemas} onFinish={handleSearch} onReset={handleReset} {...formProps} />`
   - 支持折叠模式（`collapsible` / `defaultCollapsed` / `collapsedRows`）
3. **新增** `ProQueryForm/ProQueryForm.tsx`：主组件
   - 接收 `columns` 或 `schemas`（二选一）
   - 若 `columns`，调用 `convertColumnsToSearchSchema` 转换
   - 若 `schemas`，直接使用
   - 实现 `handleSearch`：`transformSearchParams` → `beforeSearch?.(params)` → `onSearch?.(params)`
   - 实现 `handleReset`：`formRef.current?.resetFields()` → `onReset?.()`
   - **关键**：所有 hooks 提到早返回之前（修复 ProTable QueryForm 的 hooks 顺序 bug）
4. **修改** `ProTable/features/QueryForm.tsx`：改为 re-export ProQueryForm 的 4 个纯函数（保持向后兼容）
5. **新增测试** `__tests__/ProQueryForm.test.tsx`：双形态入参、onSearch/onReset、折叠模式、hooks 顺序

**收益**：MVP 可用，覆盖独立查询场景。

**风险**：中。4 个纯函数迁移需保证与原实现等价（回归测试）。

**API 草案**：

```tsx
// 形态 A：columns 列驱动（与 ProTable 互通）
<ProQueryForm
  columns={tableColumns}
  onSearch={(params) => fetchData(params)}
  onReset={() => fetchData({})}
  layout="inline"
  collapsible
  defaultCollapsed
/>

// 形态 B：schemas Schema 驱动（独立查询）
<ProQueryForm
  schemas={[
    { name: 'keyword', label: '关键词', component: 'Input', placeholder: '请输入' },
    { name: 'status', label: '状态', component: 'Select', options: statusOptions },
    { name: 'dateRange', label: '日期', component: 'DatePicker.RangePicker' },
  ]}
  onSearch={(params) => fetchChartData(params)}
  layout="inline"
/>

// 重量模式：与 ProTable DataStore 集成
const store = createDataStore<Order>();
<ProQueryForm
```

```ts
// packages/components/ProQueryForm/utils/columnsToSchema.ts（从 QueryForm.tsx 迁移）

import type { ProColumnType } from '../../ProTable/types';
import type { ProFormSchema } from '../../ProForm/types';
import { valueTypeToComponent } from './valueTypeToComponent';
import { getComponentPropsByValueType } from './columnToProps';

/**
 * 列驱动模式：ProColumnType[] → ProFormSchema[]
 * 过滤规则：hideInSearch===true / search===false / valueType==='opr' / 无 dataIndex
 */
export function convertColumnsToSearchSchema<T>(
  columns: ProColumnType<T>[],
): ProFormSchema<T>[] {
  return columns
    .filter((col) => !col.hideInSearch)
    .filter((col) => col.search !== false)
    .filter((col) => col.valueType !== 'opr')
    .filter((col) => col.dataIndex)
    .map((col) => {
      const dataIndex = col.dataIndex as string | string[];
      const name = Array.isArray(dataIndex) ? dataIndex.join('.') : dataIndex;
      const searchConfig = typeof col.search === 'object' ? col.search : {};

      return {
        name,
        label: col.title as string,
        component: searchConfig.component || (col.valueType ? valueTypeToComponent[col.valueType as keyof typeof valueTypeToComponent] : 'Input'),
        componentProps: searchConfig.componentProps || getComponentPropsByValueType(col.valueType, col),
        rules: searchConfig.rules,
        order: (searchConfig as { order?: number }).order ?? Infinity,
        ...searchConfig,
      } as ProFormSchema<T>;
    })
    .sort((a, b) => (a.order ?? Infinity) - (b.order ?? Infinity));
}

// packages/components/ProQueryForm/utils/transformParams.ts（从 QueryForm.tsx 迁移）

import type { ProColumnType } from '../../ProTable/types';

/**
 * 查询参数转换：删空值 + 逐列调用 search.transform
 */
export function transformSearchParams<T>(
  params: Record<string, unknown>,
  columns: ProColumnType<T>[],
): Record<string, unknown> {
  const result: Record<string, unknown> = {};

  for (const [key, value] of Object.entries(params)) {
    // 删空值
    if (value === null || value === undefined || value === '') continue;

    // 查找对应列的 transform
    const col = columns.find((c) => {
      const dataIndex = c.dataIndex as string | string[] | undefined;
      if (!dataIndex) return false;
      const name = Array.isArray(dataIndex) ? dataIndex.join('.') : dataIndex;
      return name === key;
    });

    const transform = col && typeof col.search === 'object' ? col.search.transform : undefined;

    if (transform) {
      const transformed = transform(value);
      if (typeof transformed === 'object' && transformed !== null) {
        Object.assign(result, transformed);
        continue;
      }
      result[key] = transformed;
    } else {
      result[key] = value;
    }
  }

  return result;
}
```

### 7.3 ProQueryForm 主组件实现要点

```tsx
// packages/components/ProQueryForm/ProQueryForm.tsx（要点节选）

import { ProForm } from '../ProForm';
import type { ProFormSchema, ProFormInstance } from '../ProForm/types';
import type { ProColumnType } from '../ProTable/types';
import { convertColumnsToSearchSchema, transformSearchParams } from './utils';
import type { ProQueryFormProps } from './types';

export const ProQueryForm = <T extends Record<string, unknown> = Record<string, unknown>>(
  props: ProQueryFormProps<T>,
) => {
  const {
    columns,
    schemas: schemasProp,
    layout = 'inline',
    column,
    collapsible,
    defaultCollapsed,
    collapsedRows,
    onSearch,
    onReset,
    beforeSearch,
    showSearch = true,
    showReset = true,
    searchButtonText,
    resetButtonText,
    store,
    urlSync,
    searchSchema,
    formProps,
    formRef,
    schemaProcessOptions,
    className,
    style,
  } = props;

  // ✅ 关键修复：所有 hooks 提到早返回之前
  const internalFormRef = useRef<ProFormInstance<T> | null>(null);
  const formInstanceRef = (formRef as React.MutableRefObject<ProFormInstance<T> | null>) || internalFormRef;

  // 形态 A：columns 转 schema；形态 B：直接使用 schemas
  const schemas = useMemo<ProFormSchema<T>[]>(() => {
    if (schemasProp) return schemasProp;
    if (columns) return convertColumnsToSearchSchema(columns);
    return [];
  }, [schemasProp, columns]);

  // URL 同步（仅重量模式）
  useUrlSyncIfNeeded({
    enabled: urlSync !== undefined && urlSync !== false && !!store,
    store,
    config: typeof urlSync === 'object' ? urlSync : undefined,
  });

  // 查询方案管理（轻量/重量模式均可）
  const searchSchemaState = useSearchSchemaIfNeeded({
    enabled: searchSchema?.enabled ?? false,
    persistenceKey: searchSchema?.persistenceKey,
    defaultSchema: searchSchema?.defaultSchema,
    initialSchemas: searchSchema?.schemas,
  });

  // ❌ 不再在早返回之后调用 hooks
  if (schemas.length === 0) return null;

  // 查询处理
  const handleSearch = useCallback(
    async (values: Record<string, unknown>) => {
      const params = transformSearchParams(values, columns ?? []);
      const finalParams = beforeSearch ? beforeSearch(params) : params;

      // 轻量模式：回调消费方
      onSearch?.(finalParams);

      // 重量模式：写入 store，自动触发订阅者
      if (store) {
        store.setQuery(finalParams);
      }
    },
    [columns, beforeSearch, onSearch, store],
  );

  // 重置处理
  const handleReset = useCallback(() => {
    formInstanceRef.current?.resetFields();
    onReset?.();
    if (store) {
      store.reset();
    }
  }, [onReset, store, formInstanceRef]);

  return (
    <div className={className} style={style}>
      {searchSchemaState && <SearchSchemaBar state={searchSchemaState} formRef={formInstanceRef} store={store} />}
      <ProForm<T>
        ref={formInstanceRef as React.RefObject<ProFormInstance<T>>}
        schemas={schemas}
        layout={layout}
        columns={column}
        collapsible={collapsible}
        defaultCollapsed={defaultCollapsed}
        collapsedRows={collapsedRows}
        onFinish={handleSearch}
        onReset={handleReset}
        showSubmitButton={showSearch}
        showResetButton={showReset}
        submitText={searchButtonText}
        resetText={resetButtonText}
        schemaProcessOptions={schemaProcessOptions}
        {...formProps}
      />
    </div>
  );
};
```

### 10.1 风险等级

| 任务 | 风险 | 主要风险点 | 缓解措施 |
| ---- | ---- | ---------- | -------- |
| MQ-0 | 低 | 仅脚手架 | — |

### 10.2 已知设计限制

1. **轻量模式不支持 useUrlSync 的完整能力**：useUrlSync 强依赖 DataStoreImpl 的 getState/setQuery/setPage 等方法，轻量模式需提供 store 适配器
   - **缓解**：实现轻量 store 适配器，或文档明确轻量模式不启用 useUrlSync
2. **重量模式继承 DataStore.setQuery 副作用**：`store.setQuery()` 会重置 page=1 + 清空选择，ProQueryForm 重量模式继承此行为
   - **缓解**：文档明确说明副作用，消费方应在 onSearch 中同步重置分页和选择
3. **useSearchSchema 的 maxCount 写死 10**：[useSearchSchema.ts](../ProTable/hooks/useSearchSchema.ts) 内部 `const maxCount = 10`，忽略配置
   - **缓解**：MQ-2 复用时顺带修复，让 maxCount 可配置
4. **formRef 单一共享**：ProTable 中 formRef 同时给 QueryForm/SearchSchemaSelector/action 用
   - **缓解**：ProQueryForm 通过 forwardRef 暴露 ProFormInstance，或通过 useProFormContext

### 10.3 回滚策略

- ProQueryForm 为**新增组件**，回滚仅需删除目录 + 移除导出，对现有组件零影响
- ProTable/features/QueryForm.tsx 的 re-export 改造可回滚（恢复原实现）
- 4 个纯函数迁移后，ProTable/features/QueryForm.tsx 改为 re-export，回滚需恢复原函数实现

### 11.2 术语表

| 术语 | 含义 |
| ---- | ---- |
| ProQueryForm | 独立查询表单组件，脱离 ProTable 使用 |
| 轻量模式 | 未传 store prop，仅管理表单状态，通过 onSearch/onReset 回调解耦 |
| 重量模式 | 传 store prop（DataStoreImpl），内部调用 store.setQuery，与 ProTable 集成 |
| 形态 A（columns） | 列驱动入参，内部 convertColumnsToSearchSchema 转 schema |
| 形态 B（schemas） | Schema 驱动入参，直接使用 ProFormSchema[] |
| QueryFormAdapter | ProTable/features/QueryForm.tsx 的兼容适配器，从 Context 取值转为 ProQueryForm props |
| 4 个纯函数 | valueTypeToComponent / getComponentPropsByValueType / convertColumnsToSearchSchema / transformSearchParams |

### 11.3 决策记录

| 决策点 | 选择 | 理由 | 日期 |
| ------ | ---- | ---- | ---- |
| 双形态入参 | columns + schemas 二选一 | columns 覆盖 ProTable 互通，schemas 覆盖独立查询 | 2026-07-04 |
| 双模式 | 轻量（onSearch 回调）+ 重量（store 集成） | 轻量覆盖 80% 独立场景，重量覆盖与 ProTable 集成的 20% | 2026-07-04 |
| ProForm 复用 | 直接 import ProForm 组件 | 复用 Schema 引擎，避免重写 | 2026-07-04 |
| useUrlSync 复用 | 直接 import（P2-1 上提后从 shared） | 与 ProTable 共用 URL 同步，避免双套实现 | 2026-07-04 |
| useSearchSchema 复用 | 直接 import（P2-1 上提后从 shared） | 与 ProTable 共用查询方案，避免双套实现 | 2026-07-04 |
| 4 个纯函数迁移 | 迁移到 ProQueryForm/utils，ProTable re-export | ProQueryForm 是独立组件，应有自己的工具；ProTable re-export 保持向后兼容 | 2026-07-04 |
| React Hooks bug 修复 | 所有 hooks 提到早返回之前 | 违反 Rules of Hooks，独立化时必须修正 | 2026-07-04 |
| ProTable QueryForm 改造 | 改为 re-export + QueryFormAdapter 适配器 | 保持公开 API 零破坏，内部切换到 ProQueryForm 实现 | 2026-07-04 |
| 不复用 useRequest | ProQueryForm 仅管查询参数，不涉及分页 | useRequest 强耦合分页/排序，与查询表单职责不符 | 2026-07-04 |
