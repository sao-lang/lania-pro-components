## 一、背景与目标

### 1.1 背景

`@lania-pro-components/components` 当前覆盖表单（ProForm）、表格（ProTable）、弹窗（ProDialog）、选择器（ProSelect）、上传（ProUpload）、操作按钮（ActionButton）六大场景，但**详情视图**这一中后台高频场景仍为空白。

详情视图通常出现在：

- **查看页**：点击列表"查看"按钮后的实体详情展示
- **侧栏摘要**：表格行展开后的内嵌摘要
- **审计/日志**：操作日志、流程轨迹的时间线展示
- **配置预览**：表单 `preview` 模式下的整体回显

目前各业务项目各自封装一层 Description 组件，存在以下问题：

- **重复封装**：label/value 布局、空值占位、复制按钮、脱敏渲染等模板代码在每个项目重复
- **与 ProTable 列定义割裂**：同一个实体的"表格列定义"和"详情字段定义"重复维护，字段类型（valueType）一致性靠人工保证
- **与 ProForm readonly 模式割裂**：ProForm 的 `readonlyRegistry` 已实现 phone/email/idCard 等脱敏渲染器，但详情视图无法复用
- **格式化逻辑重复**：金额、百分比、日期、文件大小等格式化函数已在 utils/format 沉淀，但详情视图封装层未消费

### 1.2 目标

1. **Schema 驱动**：通过 `columns` 配置数组描述详情字段，与 ProTable 列定义**互通**，同一份列定义可同时驱动表格和详情视图
2. **复用渲染器**：直接消费 ProTable 的 `renderColumnByValueType`（21+ valueType）和 ProForm 的 `readonlyRegistry`（脱敏渲染器），零拷贝
3. **多种布局**：支持 `table`（描述列表）/ `grid`（网格卡片）/ `inline`（行内）三种布局模式，覆盖不同密度场景
4. **空值/复制/脱敏开箱即用**：内置空值占位（`-`）、一键复制（消费 utils `copyToClipboard`）、脱敏渲染（消费 readonlyRegistry）
5. **与现有架构无缝衔接**：复用 utils（format/defineEnumMap/getTagColor）、复用 shared（CardContainerConfig/useResponsive）、复用 theme（light/dark）

### 1.3 范围

**包含**：

- `packages/components/ProDescriptions/` 新增组件目录
- `packages/components/ProDescriptions/transformers/` valueType → 描述项的转换器
- 与 ProTable 列定义的互通适配层

**不包含**：

- 不重新实现任何渲染器（直接消费 `renderColumnByValueType` / `readonlyRegistry`）
- 不实现复杂的单元格合并（cellMerge 强依赖 dataSource + index，不适合详情场景）
- 不修改现有 6 个组件的公开 API

## 二、现状与动机

### 2.1 现有可复用底座扫描

| 复用对象                                            | 来源                                                                                        | ProDescriptions 用途                                                                                        |
| --------------------------------------------------- | ------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| `renderColumnByValueType`                           | [ProTable/utils/columnRender.tsx](../ProTable/utils/columnRender.tsx)                       | 21+ valueType 渲染：text/number/money/percent/date/select/tag/avatar/image/link/progress/code/enum/index 等 |
| `ProColumnType`                                     | [ProTable/types.ts](../ProTable/types.ts)                                                   | 派生 `ProDescriptionColumn`（Omit 表格专有字段）                                                            |
| `readonlyRegistry`                                  | [ProForm/registry/readonlyRegistry.tsx](../ProForm/registry/readonlyRegistry.tsx)           | 脱敏渲染器：phone/email/idCard 等                                                                           |
| `defineEnumMap`                                     | [utils/src/defineEnumMap.ts](../../utils/src/defineEnumMap.ts)                              | valueEnum → label 映射                                                                                      |
| `formatNumber/formatMoney/formatPercent/formatDate` | [utils/src/format.ts](../../utils/src/format.ts)                                            | 数值/金额/百分比/日期格式化（已被 columnRender 内部消费，但 ProDescriptions 也需直接调用）                  |
| `getTagColor`                                       | [utils/src/color.ts](../../utils/src/color.ts)                                              | tag/enum 颜色映射                                                                                           |
| `copyToClipboard`                                   | [utils/src/dom.ts](../../utils/src/dom.ts)                                                  | 复制按钮（Arco Message 反馈包装器在 ProTable/utils/columnRender）                                           |
| `CardContainerConfig`                               | [ProForm/types.ts:465](../ProForm/types.ts)（P1-4 上提到 shared/types/layout.ts）           | 卡片容器配置（title/extra/bordered）                                                                        |
| `useResponsive`                                     | [ProTable/hooks/useResponsive.ts](../ProTable/hooks/useResponsive.ts)（P1-4 上提到 shared） | 响应式列数（grid 布局）                                                                                     |
| `ThemeProvider` / `useTheme`                        | theme 包                                                                                    | light/dark 适配（tag/progress 颜色）                                                                        |

**结论**：渲染底座完备，ProDescriptions 是上述能力的"轻量组合消费方"，落地成本低。

### 2.2 现有项目详情视图痛点

通过对常见业务封装的观察，重复痛点集中在：

1. **label/value 布局**：每个项目各自实现 `<dl><dt><dd>` 或 `<table>` 结构，Arco `Descriptions` 的 column 配置不灵活
2. **空值处理**：null/undefined/'' 显示什么（`-` / `空` / 不显示）每个项目不同
3. **复制能力**：长文本（订单号、UUID）需点击复制，每个项目手写一遍
4. **脱敏**：手机号/身份证/邮箱在详情视图需脱敏，与 ProForm readonly 模式逻辑重复
5. **字段类型一致性**：表格里金额显示为"¥1,234.00"，详情里却显示"1234"，靠人工保证易出错
6. **跨列合并**：某些字段（如备注）需跨多列展示，Arco `Descriptions` 的 span 配置心智重

ProDescriptions 的设计目标是**一次性解决上述六类问题**，并与 ProTable 列定义打通。

### 2.3 与 ProTable 列定义的互通难点

| 难点                                                        | 说明                                                                                                                                                                           | 解决方案                                                                |
| ----------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------- |
| `ProColumnType` 含表格专有字段                              | `oprTools` / `actions` / `proTableConfig` / `search` / `hideInSearch` / `order` / `fixed` / `width` / `align` / `ellipsis` / `colSpan` / `rowSpan` / `onCell` / `onHeaderCell` | `ProDescriptionColumn = Omit<ProColumnType, ...表格专有字段>`           |
| `renderColumnByValueType` 内部使用 `useState`               | `renderAvatar` / `renderImage` 调用 `useState` 控制预览弹窗，必须在 React 组件渲染期调用                                                                                       | ProDescriptions 不在工具函数中调用，而是封装为 `<DescriptionCell>` 组件 |
| `cellMerge` 强依赖 dataSource + index                       | 详情场景无 dataSource 概念                                                                                                                                                     | **不复用 cellMerge**，ProDescriptions 通过 `span` 字段实现跨列          |
| 列级 `render` 函数签名 `(text, record, index) => ReactNode` | 详情场景无 record/index 概念                                                                                                                                                   | ProDescriptions 调用时传 `(value, record, index=0)`，保持签名兼容       |

**结论**：通过类型 Omit + 组件封装 + 签名兼容三层适配，可零成本复用 ProTable 渲染器。

## 三、设计原则

### 3.1 列定义互通为核心

```
业务代码
    │
    ▼
  ProDescriptions（React 组件）
    │
    ▼
  ProDescriptionColumn[]（Omit ProColumnType 表格专有字段）
    │
    ▼
  renderColumnByValueType（ProTable 渲染器，零拷贝复用）
    │
    ├── valueType: text/number/money/date/...（标准类型）
    ├── readonlyRegistry（ProForm 脱敏渲染器，可选叠加）
    └── 自定义 render（逃生舱）
```

**设计原则**：同一份列定义可同时驱动 ProTable（列表）和 ProDescriptions（详情），字段类型一致性由类型系统保证。

### 3.2 三种布局模式，覆盖不同密度

| 布局              | 适用场景                 | 渲染结构                                           | 密度 |
| ----------------- | ------------------------ | -------------------------------------------------- | ---- |
| **table**（默认） | 查看页、配置预览         | `<table>` 多列描述列表（Arco `Descriptions` 风格） | 中   |
| **grid**          | 侧栏摘要、卡片内嵌       | CSS Grid 网格卡片，每项 label 上 value 下          | 低   |
| **inline**        | 行内紧凑展示、表格行展开 | `<span>label: value</span>` 横向排列               | 高   |

### 3.3 复用优于重新实现

- 渲染器：**直接 import** `renderColumnByValueType`，不重写
- 脱敏：**直接 import** `readonlyRegistry`，不重写
- 格式化：**直接 import** utils/format，不重写
- 颜色映射：**直接 import** utils/color 的 `getTagColor`
- 复制：**直接 import** ProTable/utils/columnRender 的 `copyToClipboard`（Arco Message 包装版）

### 3.4 公开 API 默认零破坏

本方案为**新增组件**，无历史 API 负担，但需承诺：

1. 一旦 v1.0 发布，`ProDescriptionsProps` / `ProDescriptionColumn` 类型默认不再做破坏性变更
2. 新增 valueType 通过 ProTable `renderColumnByValueType` 自然继承，无需 ProDescriptions 改动
3. 内部 hook 允许迭代，但导出的类型保持稳定

### 3.5 组合优于配置（与重构方案 P0-3 一致）

- ProDescriptions **不内置**性能监控浮窗
- 性能打点（`mark/measure`）由 ProDescriptions 内部产生（`descriptions-render`），监控 UI 由使用方通过 `<PerformanceMonitor>` 组合接入
- 复制按钮、脱敏开关等可选能力通过 column 配置或 props 注入，不强制启用

## 四、架构设计

### 4.1 模块划分

```
ProDescriptions
├── Core Layer（核心层）
│   ├── ProDescriptions.tsx          # 主组件
│   ├── DescriptionsContext.tsx      # 上下文（与 ProTable 多层 Context 风格一致）
│   └── types.ts                     # ProDescriptionsProps / ProDescriptionColumn 类型
│
├── Feature Layer（功能层）
│   ├── DescriptionCell.tsx          # 单项渲染（封装 renderColumnByValueType，处理 useState 陷阱）
│   ├── layouts/                     # 三种布局
│   │   ├── TableLayout.tsx          # table 布局（默认）
│   │   ├── GridLayout.tsx           # grid 布局
│   │   ├── InlineLayout.tsx         # inline 布局
│   │   └── index.ts
│   ├── CopyButton.tsx               # 复制按钮（消费 copyToClipboard）
│   └── EmptyValue.tsx               # 空值占位
│
├── Adapters（适配层）
│   └── columnAdapter.ts             # ProColumnType → ProDescriptionColumn 适配
│
└── index.ts                         # barrel
```

### 4.2 数据流

```
   ┌──────────────────────┐
   │  columns (用户传入)   │ ← ProDescriptionColumn[] 或 ProColumnType[]
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  columnAdapter       │ ← 若为 ProColumnType，Omit 表格专有字段
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  DescriptionCell     │ ← 逐项渲染
   │  ├─ valueType?       │
   │  ├─ render?          │ ← 逃生舱
   │  ├─ readonlyRegistry?│ ← 脱敏叠加
   │  └─ copyable?        │ ← 复制按钮
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  renderColumnByValueType │ ← ProTable 渲染器（零拷贝）
   │  (value, column, record, index) │
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  Layout 渲染          │ ← table / grid / inline
   └──────────────────────┘
```

### 4.3 Context 架构

与 ProTable 多层 Context 风格保持一致：

```
RootContext（全局配置层）
  ├── layout（当前布局模式）
  ├── column（默认列数）
  ├── bordered（是否带边框）
  ├── emptyText（空值占位符）
  └── theme（当前主题，从 useTheme 注入）

ColumnContext（列定义层）
  ├── columns（处理后的 ProDescriptionColumn[]）
  └── dataSource（原始数据 record）

CellContext（单项渲染层）
  ├── column（当前列定义）
  ├── value（当前值）
  ├── record（整条数据）
  └── index（字段索引）
```

## 五、与架构公共化重构方案的关联

ProDescriptions 的落地**弱依赖** [架构公共化重构方案.md](../架构公共化重构方案.md) 的部分任务。本节明确依赖关系。

| ProDescriptions 任务 | 依赖的重构方案任务                | 依赖原因                                      | 阻塞性                            |
| -------------------- | --------------------------------- | --------------------------------------------- | --------------------------------- |

### 5.2 ProDescriptions 对 shared 包的反向贡献

ProDescriptions 落地过程中**不产生**新的通用 hook（渲染器均复用现有），但反向验证了以下抽象的合理性：

| 验证对象                             | 验证方式                                                                                   |
| ------------------------------------ | ------------------------------------------------------------------------------------------ |
| `renderColumnByValueType` 的可复用性 | ProDescriptions 在非表格场景调用，验证其是否真的"渲染器"而非"表格渲染器"                   |
| `readonlyRegistry` 的可复用性        | ProDescriptions 在非表单场景调用脱敏渲染器，验证其是否真的"只读渲染器"而非"表单只读渲染器" |
| `ProColumnType` 的派生能力           | 通过 Omit 派生 `ProDescriptionColumn`，验证类型设计是否便于派生                            |
| `CardContainerConfig` 上提的必要性   | MD-3 卡片布局消费此类型，验证 P0-4 上提决策                                                |

**建议**：若 MD-2 发现 `renderColumnByValueType` 含表格语义耦合（如依赖 `column.proTableConfig`），可在重构方案 P0-5（架构债务 #10 格式化渲染器重复）中顺带修复。

### 5.3 ProDescriptions 与架构债务的协同

### MD-1：核心渲染 + table 布局（MVP）

**依赖**：MD-0

**改动范围**：

1. **新增** `ProDescriptions/DescriptionCell.tsx`：单项渲染组件
   - 接收 `column` + `value` + `record` + `index`
   - 调用 `renderColumnByValueType(value, column, record, index)`（从 `../ProTable/utils/columnRender` import）
   - 处理 `useState` 陷阱：DescriptionCell 本身是 React 组件，渲染器内部 useState 正常工作
   - 支持脱敏：若 `column.masking` 为 true，叠加 readonlyRegistry 渲染器
   - 支持复制：若 `column.copyable` 为 true，渲染 `<CopyButton>`
2. **新增** `ProDescriptions/layouts/TableLayout.tsx`：默认 table 布局
   - 基于 Arco `Descriptions` 组件封装
   - 支持 `column`（列数）/ `bordered` / `size` / `layout`（horizontal/vertical）
   - 支持 `span` 字段跨列
3. **新增** `ProDescriptions/CopyButton.tsx`：复制按钮
   - 消费 `copyToClipboard`（从 `../ProTable/utils/columnRender` import Arco 包装版）
   - 支持 `text` / `tooltip` / `icon` 配置
4. **新增** `ProDescriptions/EmptyValue.tsx`：空值占位
   - 默认显示 `-`，支持 `emptyText` 自定义
5. **新增测试** `__tests__/ProDescriptions.test.tsx`：渲染、空值、复制、span 跨列

**收益**：MVP 可用，覆盖 80% 详情场景。

**风险**：中。`renderColumnByValueType` 内部 `useState` 必须在组件渲染期调用，需测试 avatar/image 预览弹窗是否正常。

**API 草案**：

```tsx
// 基础用法
<ProDescriptions
  columns={[
    { title: '订单号', dataIndex: 'orderNo', valueType: 'text', copyable: true },
    { title: '金额', dataIndex: 'amount', valueType: 'money' },
    { title: '状态', dataIndex: 'status', valueType: 'enum', valueEnum: statusEnum },
    { title: '创建时间', dataIndex: 'createTime', valueType: 'dateTime' },
  ]}
  dataSource={{ orderNo: 'ORD20260704001', amount: 1234.5, status: 1, createTime: '2026-07-04 10:00:00' }}
/>

// 复用 ProTable 列定义
const tableColumns: ProColumnType<Order>[] = [/* ... */];
const descColumns = tableColumns.filter(c => c.valueType !== 'opr');
<ProDescriptions columns={descColumns} dataSource={order} />

// 脱敏 + 复制
<ProDescriptions
```

### 7.2 DescriptionCell 实现要点

```ts
// packages/components/ProDescriptions/DescriptionCell.tsx（要点节选）

import { renderColumnByValueType } from '../ProTable/utils/columnRender';
import { getReadonlyRenderer } from '../ProForm/registry/readonlyRegistry';
import { CopyButton } from './CopyButton';
import { EmptyValue } from './EmptyValue';
import type { DescriptionCellProps } from './types';

export const DescriptionCell: React.FC<DescriptionCellProps> = ({
  column,
  value,
  record,
  index,
  emptyText = '-',
}) => {
  // 1. 优先级：column.render > masking + readonlyRegistry > renderColumnByValueType
  let content: React.ReactNode;

  if (typeof column.render === 'function') {
    // 逃生舱：用户自定义 render
    content = column.render(value, record, index);
  } else if (column.masking && column.valueType) {
    // 脱敏渲染（消费 readonlyRegistry）
    const renderer = getReadonlyRenderer(column.valueType, 'readonly');
    content = renderer ? renderer(value, column) : renderColumnByValueType(value, column, record, index);
  } else {
    // 标准渲染（消费 ProTable 渲染器）
    // 注意：renderColumnByValueType 内部 renderAvatar/renderImage 使用 useState，
    // 必须在 React 组件渲染期调用。DescriptionCell 是组件，符合要求。
    content = renderColumnByValueType(value, column, record, index);
  }

  // 2. 空值处理
  const isEmpty = value === null || value === undefined || value === '';
  if (isEmpty && !column.render) {
    content = <EmptyValue text={emptyText} />;
  }

  // 3. 复制按钮
  const showCopy = column.copyable && !isEmpty;

  return (
    <span style={{ display: 'inline-flex', alignItems: 'center', gap: 4 }}>
      <span>{content}</span>
      {showCopy && <CopyButton text={String(value)} />}
    </span>
  );
};
```

### 7.3 columnAdapter 实现要点

```ts
// packages/components/ProDescriptions/columnAdapter.ts（要点节选）

import type { ProColumnType } from '../ProTable/types';
import type { ProDescriptionColumn } from './types';

/**
 * ProColumnType[] → ProDescriptionColumn[]
 * 自动过滤操作列和 hideInDescriptions 列
 */
export function adaptColumns<T>(columns: ProColumnType<T>[]): ProDescriptionColumn<T>[] {
  return columns
    .filter((col) => col.valueType !== 'opr')
    .filter((col) => !(col as ProDescriptionColumn<T>).hideInDescriptions)
    .map((col) => {
      // Omit 表格专有字段（通过类型系统保证，运行时仅传递 ProDescriptionColumn 允许的字段）
      const {
        oprTools,
        actions,
        proTableConfig,
        search,
        hideInSearch,
        order,
        fixed,
        width,
        align,
        ellipsis,
        colSpan,
        rowSpan,
        onCell,
        onHeaderCell,
        ...descCol
      } = col as ProColumnType<T> & ProDescriptionColumn<T>;
      return descCol as ProDescriptionColumn<T>;
    });
}
```

## 十、风险与回滚

### 10.1 风险等级

| 任务 | 风险 | 主要风险点                                     | 缓解措施                                               |
| ---- | ---- | ---------------------------------------------- | ------------------------------------------------------ |
| MD-0 | 低   | 仅脚手架                                       | —                                                      |
| MD-1 | 中   | `renderColumnByValueType` 内部 `useState` 陷阱 | DescriptionCell 封装为组件，确保渲染器在组件渲染期调用 |
| MD-2 | 中   | ProColumnType 新增 `hideInDescriptions` 字段   | 仅新增可选字段，零破坏；ProTable 默认忽略此字段        |
| MD-3 | 低   | grid 布局响应式列数                            | 弱依赖 P0-4，可先用 ProTable 版 useResponsive          |

### 10.2 已知设计限制

1. **`renderColumnByValueType` 的 useState 耦合**：avatar/image 预览弹窗依赖 useState，ProDescriptions 必须通过组件封装调用，不能在工具函数中调用
   - **缓解**：DescriptionCell 是 React 组件，符合 useState 调用要求
2. **cellMerge 不复用**：cellMerge 强依赖 dataSource + index，详情场景无此概念
   - **缓解**：ProDescriptions 通过 `span` 字段实现跨列，不处理行列合并
3. **脱敏渲染器依赖 ProForm readonlyRegistry**：若 ProForm 未初始化 registry，脱敏渲染降级为标准渲染
   - **缓解**：文档明确说明，并提供 `maskingFallback` 配置

### 10.3 回滚策略

- ProDescriptions 为**新增组件**，回滚仅需删除目录 + 移除导出，对现有组件零影响
- ProTable 新增的 `hideInDescriptions` 字段即使回滚 ProDescriptions，也可保留（对未来其他详情场景有用）
- 不涉及 shared 包新增内容，无需回滚 shared

### 10.4 灰度策略

- MD-1 完成后在内部项目试用 1-2 周，验证 21+ valueType 渲染一致性
- MD-2 完成后再次灰度，验证与 ProTable 列定义的互通场景
- 正式发布前完成 docs 示例 + README
