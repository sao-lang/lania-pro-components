## 一、背景与目标

### 1.1 背景

`@lania-pro-components/components` 当前覆盖表单、表格、弹窗、选择器、上传、按钮六大场景，但**页面级布局**这一中后台最顶层的骨架仍为空白。

中后台典型页面结构：

```
┌─────────────────────────────────────────────────────────┐
│  PageHeader（标题/副标题/描述/面包屑/extra 操作区）        │
├──────────┬──────────────────────────────────────────────┤
│          │                                              │
│  Sider   │  Content（业务内容区，常包裹在 Card 内）       │
│ (可选)   │                                              │
│          │                                              │
├──────────┴──────────────────────────────────────────────┤
│  Footer（底部按钮区：提交/取消/重置，位置左/中/右）        │
└─────────────────────────────────────────────────────────┘
```

目前各业务项目各自实现这套布局，存在以下问题：

- **重复封装**：PageHeader 的 title/subTitle/description/extra、Footer 的按钮位置（left/center/right）每个项目重复实现
- **响应式断点不统一**：移动端/平板/桌面端布局切换逻辑散落，部分页面用 CSS media query，部分用 JS 监听 window resize
- **主题联动缺失**：Arco theme 切换时，自定义布局组件的硬编码颜色（背景、边框）不跟随
- **与现有组件割裂**：ProTable 的 `Toolbar` 已有 title/subTitle/description 三件套，ProForm 的 `cardContainer` 已有卡片容器配置，ProDialog 已有 FooterPosition，但页面级布局未消费这些既有抽象
- **Arco PageHeader 未被项目采用**：项目全文检索 `PageHeader` 零匹配，说明 Arco 内置组件不满足业务需求，需自定义

### 1.2 目标

1. **统一页面骨架**：通过 `ProLayout` 组件封装 PageHeader + Content + Footer + Sider 四个区域，提供一致的页面结构
2. **复用既有抽象**：消费 ProTable `Toolbar` 的 title/subTitle/description 三件套模式、ProForm `CardContainerConfig` 卡片容器配置、ProDialog `FooterPosition` + `getFooterJustify` 底部按钮位置
3. **响应式内置**：消费 `useResponsive`（P1-4 上提到 shared）实现 mobile/tablet/desktop 三档断点自动切换布局模式
4. **主题联动**：消费 theme 包的 `ThemeProvider` / `useTheme`，布局颜色（背景/边框/阴影）自动跟随 light/dark
5. **多种布局模式**：支持 `top`（顶栏）/ `side`（侧栏）/ `mix`（混合）三种布局模式，覆盖不同业务场景

### 1.3 范围

**包含**：

- `packages/components/ProLayout/` 新增组件目录
- `packages/components/ProLayout/layouts/` 三种布局模式实现
- `packages/components/ProLayout/components/` PageHeader / Footer / Sider 子组件

**不包含**：

- 不实现路由系统（仅提供布局容器，路由由消费方自行接入）
- 不实现菜单配置（Sider 仅提供容器，菜单内容由 children 注入）
- 不实现权限管理（权限由消费方在 children 中自行处理）
- 不修改现有 6 个组件的公开 API

## 二、现状与动机

### 2.1 现有可复用底座扫描

| 复用对象 | 来源 | ProLayout 用途 |
| -------- | ---- | -------------- |
| `useResponsive` | [ProTable/hooks/useResponsive.ts](../ProTable/hooks/useResponsive.ts)（P1-4 上提到 shared） | 响应式断点：isMobile/isTablet/isDesktop，触发布局模式切换 |
| `useResponsiveColumns` | [ProTable/hooks/useResponsive.ts](../ProTable/hooks/useResponsive.ts) | Sider 折叠宽度自适应 |
| `ThemeProvider` / `useTheme` / `lightTheme` / `darkTheme` / `ThemeType` | [theme/src/index.ts](../../theme/src/index.ts) | light/dark 主题联动，布局颜色跟随 |
| `CardContainerConfig` | [ProForm/types.ts:465](../ProForm/types.ts)（P0-4 上提到 shared/types/layout.ts） | Content 区卡片容器配置（title/extra/bordered/bodyStyle） |
| `LayoutMode` | [ProForm/types.ts:36](../ProForm/types.ts)（P0-4 上提到 shared/types/layout.ts） | `'horizontal' \| 'vertical' \| 'inline' \| 'compact'`，复用为 Content 区布局方向 |
| `FooterPosition` | [ProDialog/types.ts:34](../ProDialog/types.ts) | Footer 按钮位置：`'left' \| 'center' \| 'right'` |
| `getFooterJustify` | [ProDialog/utils.ts:51](../ProDialog/utils.ts) | FooterPosition → CSS justify-content 映射 |
| `ProTableToolbarConfig` | [ProTable/features/Toolbar.tsx](../ProTable/features/Toolbar.tsx) | PageHeader 的 title/subTitle/description 三件套模式参考 |
| `ProTableBatchOperationConfig` | [ProTable/features/BatchOperation.tsx](../ProTable/features/BatchOperation.tsx) | 批量操作条布局参考（Alert 容器 + 居中布局） |

**结论**：布局相关抽象散落在 ProForm / ProTable / ProDialog / theme 四处，ProLayout 是这些抽象的"页面级组合消费方"，落地能反向验证 P0-4 / P1-4 上提决策的合理性。

### 2.2 现有项目页面布局痛点

通过对常见业务页面的观察，重复痛点集中在：

1. **PageHeader 重复实现**：title/subTitle/description/breadcrumb/extra 的布局每个项目写一遍，与 ProTable Toolbar 的三件套模式重复但不互通
2. **Footer 按钮位置不统一**：有的项目用 flex space-between，有的用 margin auto，有的用 Arco ButtonGroup，与 ProDialog 的 FooterPosition + getFooterJustify 重复但不互通
3. **响应式断点混乱**：部分页面用 CSS `@media (max-width: 768px)`，部分用 `window.matchMedia`，部分用 `useResponsive`，三套实现并存
4. **主题颜色硬编码**：自定义布局组件常用 `style={{ background: '#fff' }}`，dark 主题下不跟随
5. **Sider 折叠状态管理**：侧边栏折叠/展开状态散落在各页面 useState，未持久化（刷新丢失）
6. **Content 区卡片包裹重复**：每个页面手动 `<Card title="..." extra="..."><Content /></Card>`，与 ProForm cardContainer 配置重复但不互通

ProLayout 的设计目标是**一次性解决上述六类问题**，并通过组合消费既有抽象，反向推动 P0-4 / P1-4 上提。

### 2.3 与现有组件抽象的互通难点

| 难点 | 说明 | 解决方案 |
| ---- | ---- | -------- |
| `CardContainerConfig` 是 ProForm 内联联合类型 | [ProForm/types.ts:465](../ProForm/types.ts) 未独立导出，仅在 ProFormProps 内联使用 | P0-4 上提到 shared/types/layout.ts，ProLayout 直接 import |
| `LayoutMode` 含 ProForm 语义（compact） | `'horizontal' \| 'vertical' \| 'inline' \| 'compact'` 的 `compact` 是 ProForm 紧凑表单专用 | ProLayout 的 Content 布局仅消费 `'horizontal' \| 'vertical' \| 'inline'`，`compact` 由 ProForm 内部保留 |
| `FooterPosition` 是 ProDialog 内部类型 | [ProDialog/types.ts:34](../ProDialog/types.ts) 未上提 | ProLayout 直接 import，或 P0-4 一并上提到 shared/types/layout.ts |
| `useResponsive` 返回值含 ProTable 语义（columns） | `useResponsiveColumns` 是为表格列数设计，但 `isMobile/isTablet/isDesktop` 是通用的 | ProLayout 仅消费 `isMobile/isTablet/isDesktop`，不消费 `columns` |
| Arco `PageHeader` 项目未使用 | 项目全文检索 `PageHeader` 零匹配，说明 Arco 内置组件不满足业务需求 | ProLayout 自定义 `PageHeader` 子组件，参考 ProTable Toolbar 三件套模式 |

**结论**：通过 P0-4 上提 `CardContainerConfig` / `LayoutMode` / `FooterPosition` 到 shared/types/layout.ts，ProLayout 可零成本消费；过渡期可先从 ProForm/ProDialog 直接 import。

## 三、设计原则

### 3.1 组合既有抽象为核心

```
业务代码
    │
    ▼
  ProLayout（React 组件）
    │
    ├── PageHeader（消费 ProTable Toolbar 三件套模式）
    │   ├── title / subTitle / description
    │   ├── breadcrumb（Arco Breadcrumb）
    │   └── extra（操作按钮区）
    │
    ├── Sider（可选，消费 useResponsive 折叠）
    │   ├── collapsed（持久化到 localStorage）
    │   └── children（菜单内容由消费方注入）
    │
    ├── Content（消费 CardContainerConfig）
    │   ├── cardContainer（卡片容器配置）
    │   └── children（业务内容）
    │
    └── Footer（消费 FooterPosition + getFooterJustify）
        ├── position（left/center/right）
        └── children（按钮组）
```

**设计原则**：ProLayout 是既有抽象的组合消费方，不重新发明任何布局概念。

### 3.2 三种布局模式，覆盖不同场景

| 布局模式 | 适用场景 | 结构 | 响应式行为 |
| -------- | -------- | ---- | ---------- |
| **top**（默认） | 简单页面、详情页 | PageHeader + Content + Footer 垂直排列 | mobile 不变 |
| **side** | 后台管理主页面 | Sider + (PageHeader + Content + Footer) | mobile 自动折叠 Sider（drawer 模式） |
| **mix** | 复杂中后台 | top 布局 + Sider 在 Content 区域内 | mobile 自动转 top |

### 3.3 复用优于重新实现

- 卡片容器：**直接 import** `CardContainerConfig` 类型（P0-4 上提后从 shared）
- 底部位置：**直接 import** `FooterPosition` 类型 + `getFooterJustify` 函数
- 响应式：**直接 import** `useResponsive`（P1-4 上提后从 shared）
- 主题：**直接消费** theme 包的 `useTheme`
- PageHeader 三件套：**参考** ProTable Toolbar 模式，不直接 import（避免 ProTable 耦合）

### 3.4 公开 API 默认零破坏

本方案为**新增组件**，无历史 API 负担，但需承诺：

1. 一旦 v1.0 发布，`ProLayoutProps` / `PageHeaderProps` / `FooterProps` / `SiderProps` 类型默认不再做破坏性变更
2. 新增布局模式通过扩展点支持，不修改核心接口
3. 内部 hook 允许迭代，但导出的类型保持稳定

### 3.5 组合优于配置（与重构方案 P0-3 一致）

- ProLayout **不内置**性能监控浮窗
- 性能打点（`mark/measure`）由 ProLayout 内部产生（`layout-render`），监控 UI 由使用方通过 `<PerformanceMonitor>` 组合接入
- 菜单、路由、权限等业务能力通过 children / render prop 注入，不通过 props 配置驱动

## 四、架构设计

### 4.1 模块划分

```
ProLayout
├── Core Layer（核心层）
│   ├── ProLayout.tsx                # 主组件
│   ├── LayoutContext.tsx            # 上下文（与 ProTable 多层 Context 风格一致）
│   └── types.ts                     # ProLayoutProps / PageHeaderProps / FooterProps / SiderProps
│
├── Component Layer（子组件层）
│   ├── PageHeader.tsx               # 页头（title/subTitle/description/breadcrumb/extra）
│   ├── Footer.tsx                   # 底部按钮区（消费 FooterPosition + getFooterJustify）
│   ├── Sider.tsx                    # 侧边栏（折叠状态持久化）
│   └── Content.tsx                  # 内容区（消费 CardContainerConfig）
│
├── Layout Mode Layer（布局模式层）
│   ├── layouts/
│   │   ├── TopLayout.tsx            # top 布局（默认）
│   │   ├── SideLayout.tsx           # side 布局
│   │   ├── MixLayout.tsx            # mix 布局
│   │   └── index.ts
│
├── Hooks（内部 hook）
│   └── useSiderCollapsed.ts         # Sider 折叠状态（localStorage 持久化）
│
└── index.ts                         # barrel
```

### 4.2 数据流

```
   ┌──────────────────────┐
   │  ProLayoutProps      │ ← layout / header / sider / content / footer / responsive
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  useResponsive       │ ← 消费 P1-4 shared hook
   │  → isMobile/isTablet │
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  Layout 模式选择      │ ← top / side / mix，mobile 自动转 top
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  PageHeader 渲染      │ ← 消费 ProTable Toolbar 三件套模式
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  Sider + Content 渲染 │ ← Sider 折叠状态持久化，Content 包裹 CardContainerConfig
   └──────────┬───────────┘
              │
              ▼
   ┌──────────────────────┐
   │  Footer 渲染          │ ← 消费 FooterPosition + getFooterJustify
   └──────────────────────┘
```

### 4.3 Context 架构

与 ProTable 多层 Context 风格保持一致：

```
RootContext（全局配置层）
  ├── layout（当前布局模式）
  ├── theme（当前主题，从 useTheme 注入）
  ├── responsive（useResponsive 返回值）
  └── collapsed（Sider 折叠状态 + toggleCollapsed）

HeaderContext（页头层）
  ├── title / subTitle / description
  ├── breadcrumb
  └── extra

ContentContext（内容层）
  ├── cardContainer（CardContainerConfig）
  └── children

FooterContext（底部层）
  ├── position（FooterPosition）
  └── children
```

## 五、与架构公共化重构方案的关联

ProLayout 的落地**弱依赖** [架构公共化重构方案.md](../架构公共化重构方案.md) 的部分任务。本节明确依赖关系。

### 5.1 ProLayout 对重构方案的依赖

| ProLayout 任务 | 依赖的重构方案任务 | 依赖原因 | 阻塞性 |
| -------------- | ----------------- | -------- | ------ |
| ML-0 脚手架 | 无 | 仅新增目录，无 shared 依赖 | 无阻塞 |
| ML-1 核心 | **P1-4 useResponsive 上提** | 响应式断点检测 | 弱阻塞（可先用 ProTable 版） |
| ML-1 核心 | **P0-4 CardContainerConfig 上提** | Content 区卡片容器配置类型 | 弱阻塞（可先用 ProForm 内联类型） |
| ML-1 核心 | **P0-4 FooterPosition 上提** | Footer 按钮位置类型 | 弱阻塞（可先用 ProDialog 内联类型） |
| ML-2 Sider + 主题 | theme 包（已存在） | light/dark 主题联动 | 无阻塞 |

**结论**：ProLayout 可以**立即启动**，ML-1 弱依赖 P0-4 / P1-4，可先用 ProForm/ProDialog/ProTable 内联类型作为过渡，待 P0-4 / P1-4 完成后切换 import 路径。

### 5.2 ProLayout 对 shared 包的反向贡献

ProLayout 落地过程中**不产生**新的通用 hook，但反向验证了以下抽象的合理性：

| 验证对象 | 验证方式 |
| -------- | -------- |
| `CardContainerConfig` 上提的必要性 | ProLayout Content 区消费此类型，验证 P0-4 上提决策 |
| `FooterPosition` + `getFooterJustify` 上提的必要性 | ProLayout Footer 区消费此类型 + 函数，验证 P0-4 上提决策 |
| `useResponsive` 上提的必要性 | ProLayout 响应式布局消费此 hook，验证 P1-4 上提决策 |
| `LayoutMode` 上提的必要性 | ProLayout Content 布局方向消费此类型（部分子集），验证 P0-4 上提决策 |

**建议**：若 ML-1 发现 `CardContainerConfig` / `FooterPosition` 类型定义需要调整（如字段缺失），可在 P0-4 任务中顺带完善。

### 5.3 ProLayout 与架构债务的协同

| 重构方案债务 | ProLayout 落地时是否触及 | 协同方式 |
| ----------- | ------------------------ | -------- |
| #5 Modal.confirm Promise 包装重复 | 无 | 不涉及 |
| #8 ProForm performance.monitor 配置耦合 | 无 | 不涉及 |

### 5.4 在重构方案任务清单中的位置

**收益**：建立目录骨架与类型契约，为后续任务提供编译验证基础。

**风险**：低。仅脚手架，无运行时逻辑。

### ML-1：核心 PageHeader + Content + Footer + 响应式（MVP）

**依赖**：ML-0、P1-4（弱）、P0-4（弱）

**改动范围**：

1. **新增** `ProLayout/components/PageHeader.tsx`：页头组件
   - 消费 ProTable Toolbar 三件套模式：`title` / `subTitle` / `description`
   - 支持 `breadcrumb`（Arco `Breadcrumb` 组件）
   - 支持 `extra`（操作按钮区，右对齐）
   - 布局：`display: flex; justify-content: space-between; align-items: center`
   - 主题联动：背景色用 `var(--color-bg-2)`，边框用 `var(--color-border-2)`
2. **新增** `ProLayout/components/Content.tsx`：内容区组件
   - 消费 `CardContainerConfig`（P0-4 上提后从 shared import，过渡期从 `../ProForm/types` import）
   - 支持 `cardContainer` 配置：`title` / `extra` / `bordered` / `bodyStyle`
   - 支持 `LayoutMode`：`'horizontal' | 'vertical' | 'inline'`（不含 `compact`，由 ProForm 内部保留）
   - 不传 `cardContainer` 时，直接渲染 children（无卡片包裹）
3. **新增** `ProLayout/components/Footer.tsx`：底部按钮区组件
   - 消费 `FooterPosition`（`'left' | 'center' | 'right'`，从 `../ProDialog/types` import，P0-4 上提后从 shared import）
   - 消费 `getFooterJustify(position)`（从 `../ProDialog/utils` import）
   - 固定在底部（`position: sticky; bottom: 0`），背景色 `var(--color-bg-1)`

### 7.4 Sider + useSiderCollapsed 实现要点

```ts
// packages/components/ProLayout/hooks/useSiderCollapsed.ts（要点节选）

import { useState, useEffect, useCallback } from 'react';

export function useSiderCollapsed(options: {
  collapsed?: boolean;          // 受控值
  onCollapsedChange?: (c: boolean) => void;
  storageKey?: string;
}): [boolean, (c: boolean) => void] {
  const { collapsed: controlled, onCollapsedChange, storageKey } = options;
  const isControlled = controlled !== undefined;

  // 初始值：受控优先，否则从 localStorage 读取，否则 false
  const [internalCollapsed, setInternalCollapsed] = useState<boolean>(() => {
    if (isControlled) return controlled!;
    if (storageKey && typeof window !== 'undefined') {
      return window.localStorage.getItem(`pro-layout-sider-${storageKey}`) === 'true';
    }
    return false;
  });

  // 受控模式同步
  useEffect(() => {
    if (isControlled) setInternalCollapsed(controlled!);
  }, [isControlled, controlled]);

  // 持久化
  useEffect(() => {
    if (!storageKey || typeof window === 'undefined') return;
    window.localStorage.setItem(`pro-layout-sider-${storageKey}`, String(internalCollapsed));
  }, [internalCollapsed, storageKey]);

  const setCollapsed = useCallback((c: boolean) => {
    if (!isControlled) setInternalCollapsed(c);
    onCollapsedChange?.(c);
  }, [isControlled, onCollapsedChange]);

  return [internalCollapsed, setCollapsed];
}
```

## 十、风险与回滚

### 10.1 风险等级

| 任务 | 风险 | 主要风险点 | 缓解措施 |
| ---- | ---- | ---------- | -------- |
| ML-0 | 低 | 仅脚手架 | — |
| ML-1 | 低 | 纯 UI 渲染，复用既有抽象 | 弱依赖 P0-4/P1-4，可先用内联类型过渡 |

### 10.2 已知设计限制

1. **不内置菜单系统**：Sider 仅提供容器，菜单内容由 children 注入（Arco Menu 由消费方自行接入）
   - **缓解**：文档提供 Menu 集成示例
2. **不内置路由**：ProLayout 仅提供布局容器，路由由消费方自行接入（react-router / Next.js 等）
   - **缓解**：文档提供 react-router v6 集成示例
3. **LayoutMode 部分消费**：`'compact'` 是 ProForm 紧凑表单专用，ProLayout 不消费此模式
   - **缓解**：类型定义中 Content.layout 仅暴露 `'horizontal' | 'vertical' | 'inline'`
4. **Arco PageHeader 未采用**：项目全文检索 `PageHeader` 零匹配，ProLayout 自定义 PageHeader 子组件
   - **缓解**：参考 ProTable Toolbar 三件套模式，保持一致性

### 10.3 回滚策略

- ProLayout 为**新增组件**，回滚仅需删除目录 + 移除导出，对现有组件零影响
- 不涉及 shared 包新增内容，无需回滚 shared
- 不修改现有 6 个组件的任何代码，无需回滚其他组件

### 10.4 灰度策略

- ML-1 完成后在内部项目试用 1-2 周，验证 top 布局 + 响应式行为
- ML-2 完成后再次灰度，验证 side/mix 布局 + Sider 折叠持久化

### 11.3 决策记录

| 决策点 | 选择 | 理由 | 日期 |
| ------ | ---- | ---- | ---- |
| 布局模式 | top / side / mix 三种 | 覆盖简单页面、后台管理、复杂中后台三种场景 | 2026-07-04 |
| PageHeader 实现 | 自定义（不复用 Arco PageHeader） | 项目零匹配，Arco 内置不满足业务需求；参考 ProTable Toolbar 三件套 | 2026-07-04 |
| Footer 复用 | 直接 import ProDialog 的 FooterPosition + getFooterJustify | 与弹窗底部行为一致，避免重复 | 2026-07-04 |
| Content 复用 | 直接 import ProForm 的 CardContainerConfig | 与表单卡片容器一致，避免重复 | 2026-07-04 |
| LayoutMode 部分消费 | 仅 'horizontal' / 'vertical' / 'inline' | 'compact' 是 ProForm 紧凑表单专用，页面级布局不适用 | 2026-07-04 |
| 响应式 | 消费 useResponsive（P1-4 上提） | 与 ProTable 共用断点检测，避免双套实现 | 2026-07-04 |
| Sider 折叠持久化 | localStorage | 刷新保持状态，提升体验；支持多实例 storageKey 区分 | 2026-07-04 |
| 不内置菜单/路由 | 仅提供容器，children 注入 | 保持组件边界清晰，避免与路由库耦合 | 2026-07-04 |
| 主题联动 | CSS 变量 var(--color-*) | 自动跟随 Arco theme，无需 JS 监听 | 2026-07-04 |
